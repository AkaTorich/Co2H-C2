# scelot

Преобразователь Windows-PE (EXE/DLL, native x86/x64 и сборки .NET) в
позиционно-независимый шелл-код. Концептуально — аналог open-source
проекта Donut, но написан с нуля на C / WinAPI под MSVC, без CRT в
загрузчике.

Уникальная особенность — полиморфный stub: при каждой генерации
байтовое представление загрузчика отличается, при этом семантика
сохраняется.

## Возможности

- Native PE x64 и x86 (ручной маппинг с релокациями, импортами и правами секций).
- DLL: вызов `DllMain(DLL_PROCESS_ATTACH)` и указанного экспорта (`-e Name`).
- .NET в памяти через `ICorRuntimeHost` + `_AppDomain::Load_3` (без сброса на диск).
- AES-128-CTR шифрование Instance и нагрузки разными случайными IV.
- Резолв API через PEB-walk и хеши имён, без читаемых строк в `.rdata`.
- Полиморфные правки stub:
  - случайный AES-ключ + случайная XOR-маска ключа,
  - случайный IV (длина 16 байт),
  - случайная отбойная подушка 4–67 байт между stub и Instance.

## Структура проекта

```
scelot/
  generator/   исходники scelot.exe (CLI, разбор PE, AES, RNG, полиморфизм)
  loader/      позиционно-независимый stub (компилируется в плоский .text)
  build/       build_all.cmd и вспомогательные скрипты
```

## Сборка

Открыть **x64 Native Tools Command Prompt for VS** (или `vcvarsall.bat x64`)
и из корня проекта запустить:

```
build\build_all.cmd
```

Скрипт:
1. собирает loader stub под x64 → `loader\stub_x64.bin`,
2. собирает loader stub под x86 → `loader\stub_x86.bin`,
3. перегенерирует `generator\embed.c` из свежих `.bin`,
4. собирает `scelot.exe`.

## Тесты

В `tests/payloads/` четыре эталонных нагрузки:

- `hello_msg/` — `MessageBoxA("scelot OK")`, минимум импортов;
- `args_echo/` — консольный EXE, печатает `argv` (проверка хука `GetCommandLine`);
- `dll_run/` — DLL с экспортом `Run(LPSTR)`;
- `net_hello/` — .NET-консольник `Console.WriteLine` (проверка in-memory CLR-хостинга).

Сквозной прогон:

```
tests\run_tests.cmd
```

Скрипт собирает все четыре нагрузки, генерирует для каждой `.bin` +
launcher через `scelot.exe`, запускает, затем регенерирует
`hello_msg.bin` пять раз подряд и печатает SHA-256 каждой версии —
все пять хешей должны различаться, и каждая копия должна корректно
запускаться.

## Использование

```
scelot <input.exe|.dll> -o out.bin [опции]

  -a x86|x64                 разрядность (по умолчанию — как у входа)
  -e <name>                  имя экспорта (для DLL-нагрузки)
  -c <Namespace.Class>       класс .NET
  -m <Method>                метод .NET
  --args "..."               аргументы командной строки для нагрузки
  --exit exit|thread|return  поведение после возврата (по умолчанию exit)
  --exe <path>               собрать тестовый launcher EXE
```

Примеры:

```
scelot test\hello.exe -o hello.bin --exe hello_run.exe
hello_run.exe hello.bin

scelot mydll.dll -o run.bin -e Run --args "arg1 arg2"

scelot Demo.exe -o demo.bin   # .NET — в памяти, без диска
```

## Контракт шелл-кода

Шелл-код вызывается как `void()` — без аргументов. Самолокализующийся
пролог (написан на ml64/ml, размещён в самом начале `.text`) через
`call/pop` вычисляет собственный адрес и передаёт его C-стороне.
Внутри пролога находится сигнатурный регион для структурного
полиморфизма — генератор перезаписывает его случайной валидной
последовательностью многобайтовых NOP-эквивалентов.

## Внутренний формат блоба

```
[ stub               ] — позиционно-независимый загрузчик (сюда передаётся управление)
[ random pad 4..67   ] — отбойные байты для полиморфизма
[ Instance (AES-CTR) ] — метаданные, тип нагрузки, IV нагрузки, аргументы
[ Payload (AES-CTR)  ] — исходный PE целиком
```

Внутри stub в плоский `.text` зашиты три плейсхолдера:
- 32-битный размер «stub + pad» (поиск по сигнатуре `0xA5A5A501`),
- 16-байтовый ключ XOR-маска (`F0 0D BE EF...`),
- 16-байтовый замаскированный AES-ключ (`C0 DE C0 DE...`),
- 16-байтовый AES-IV для Instance (`BA AD F0 0D...`).

Генератор находит эти сигнатуры в `stub_<arch>.bin` и переписывает их
случайными значениями при каждой сборке шелл-кода.

## Известные ограничения

- **.NET-нагрузки** запускаются через `CreateProcess`: stub записывает сборку
  во временный `.exe` в `%TEMP%`, запускает её и удаляет файл после завершения.
  Windows активирует CLR автоматически через стандартный shim, никаких
  специальных требований к сигнатуре `Main` нет — `static void Main(string[])`
  работает как есть. Архитектура (x86/x64) сборки должна совпадать с архитектурой
  stub'а; для .NET Framework сборок это значит компилировать с `csc /platform:x64`.
  Минус подхода — на время выполнения существует артефакт на диске.
- **`argc`/`argv` в нагрузке через CRT**: IAT-хук `GetCommandLineA/W`
  работает для прямых вызовов, но стандартный `mainCRTStartup` парсит
  командную строку напрямую из `PEB->ProcessParameters->CommandLine`,
  обходя IAT. Чтобы аргументы доходили до `int main(argc, argv)`, надо
  патчить `RTL_USER_PROCESS_PARAMETERS.CommandLine`. Сейчас нагрузки без
  CRT (см. `tests/payloads/`) видят корректные аргументы через прямой
  вызов `GetCommandLineA`.

## Реализовано в текущей версии

- Самолокализующийся пролог на MASM (`prologue_x64.asm`, `prologue_x86.asm`)
  — шелл-код вызывается без аргументов.
- Структурный полиморфизм: регионы между сигнатурами
  `CC CC DE AD <id>` ... `CC CC BE EF <id>` перезаписываются случайной
  валидной последовательностью многобайтовых NOP-эквивалентов
  (1, 2, 3, 4, 5, 6, 7, 8, 9 байт — рекомендованные кодировки Intel SDM)
  с точным сохранением длины региона.
- Хук `GetCommandLineA` / `GetCommandLineW` через правку IAT нагрузки
  (kernel32.dll, KERNELBASE.dll, api-ms-win-core-processenvironment-l1-1-0.dll).
  Аргументы из `--args` хранятся внутри stub и доступны нагрузке через
  стандартные функции CRT.
- `RtlAddFunctionTable` для x64-нагрузок с непустым `.pdata` —
  корректно работает SEH/C++-исключения.

## Возможные дальнейшие шаги

- Перестановка независимых базовых блоков (требует встроенного
  дизассемблера для правки относительных смещений после перестановки).
- Замена пар регистров на эквивалентные по семантике (`mov rcx,rax` ↔
  `lea rcx,[rax]` ↔ `push rax; pop rcx`) с подбором кодировок одной длины.
- Обход `_AppDomain::Load_3` через прямую инжекцию байт-кода в обход
  COM-маршалинга — снижает количество видимых импортов из mscoree.
