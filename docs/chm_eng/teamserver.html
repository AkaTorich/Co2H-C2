Teamserver

The teamserver is launched from the client via the Start Server button on the toolbar. Configuration (TLS certificates, listeners, database) is stored in configs/srv.toml alongside the client and edited through the GUI; normally you do not need to run teamserver.exe directly.

Config Structure: srv.toml

General Sections

[operators]
bind = "0.0.0.0"
port = 50050

[tls]
cert = "certs/server.crt"
key  = "certs/server.key"
ca   = "certs/ca.crt"      # comment out to disable mTLS (development only)

[database]
path = "co2h.db"

[io]
threads = 0                 # 0 = auto (based on number of cores)

[log]
dir = "logs"

Listener Configuration

Each listener is described by a [[listeners]] section. Multiple sections can be specified — all will start when the teamserver starts.

HTTPS listener

[[listeners]]
kind         = "https"
name         = "https-main"
bind         = "0.0.0.0"
port         = 443
tls_cert     = "certs/listener.crt"
tls_key      = "certs/listener.key"
profile      = "profiles/default.toml"   # malleable profile (optional)
# listener_key = "aabb...64 hex chars"   # optional; omit for auto

TCP (raw) listener

TCP listeners are usually created dynamically through the client UI. If necessary, they can be described in the config:

[[listeners]]
kind         = "tcp"
name         = "tcp-internal"
bind         = "0.0.0.0"
port         = 4444
# listener_key = "aabb...64 hex chars"

SMB (named pipe) listener

SMB listeners are also created dynamically. Example for the config:

[[listeners]]
kind         = "smb"
name         = "smb-lateral"
bind         = "0.0.0.0"
port         = 445
# listener_key = "aabb...64 hex chars"

DNS (UDP) listener

[[listeners]]
kind         = "dns"
name         = "dns-c2"
bind         = "0.0.0.0"
port         = 53
c2_domain    = "c2.example.com"
# listener_key = "aabb...64 hex chars"
ParameterDescription
c2_domainAuthoritative subdomain delegated to the teamserver's IP via an NS record
portUDP port (standard — 53)

Multiple listeners simultaneously

You can run HTTPS and DNS at the same time — each with its own key:

[[listeners]]
kind         = "https"
name         = "https-main"
bind         = "0.0.0.0"
port         = 443
tls_cert     = "certs/listener.crt"
tls_key      = "certs/listener.key"

[[listeners]]
kind         = "dns"
name         = "dns-backup"
bind         = "0.0.0.0"
port         = 53
c2_domain    = "c2.example.com"

The listener_key Parameter

AES-256 listener key (64 hex chars). Key selection logic on startup:

  1. If listener_key is specified in the config — it is used as-is and saved to the DB.
  2. Otherwise — the key is taken from the DB (saved from the previous run).
  3. If not in the DB — a random key is generated and saved.

Use case: when a beacon is configured for automatic failover between servers (fallback C2), all servers in the chain must use the same listener key.

To copy the current listener key: Listeners tab → right-click → Copy listener key.

What the Teamserver Does on Startup

  1. Reads the config and initializes the network I/O subsystem.
  2. Opens the database and migrates the schema.
  3. Brings up the operator connection port (mTLS).
  4. Starts all listeners declared in [[listeners]] sections.
  5. Generates a key for each listener and displays it in the client's Listeners tab.

Operator Authentication

Operator passwords are stored in the DB as a hash. The default admin is created on first initialization:

usernameadmin
passwordchangeme
Changing the password is the mandatory first operation in a real deployment.

Credentials Store

The shared credentials store for all operators. Stored in the teamserver database; changes are automatically distributed to all connected operators — the Credentials tab on each client updates without reloading.

Supported record types: password, hash, ticket, identity. Management — via the Credentials tab in the client: Refresh, Add, Delete.

Starting from the Client

Toolbar → Start Server. The process stdout/stderr automatically goes to the Logs tab; the status (running / stopped) and listener port are shown in the client status bar. Stop Server shuts the process down gracefully.

Standalone Launch (without the Client)

The teamserver can be run independently of the client — for example, on a remote VPS or in a Docker container. The only required argument is the path to the config:

Windows

teamserver.exe --config C:\path\to\srv.toml

Linux

./teamserver --config /opt/co2h/srv.toml

If the --config argument is not specified, the teamserver looks for configs/srv.toml in the current working directory.

Additional Arguments

ArgumentDescriptionDefault
--config <path>Path to the TOML configconfigs/srv.toml
--bind <addr>Bind address for the operator port (overrides config)from config
--port <num>Port for operator connections (overrides config)from config
--verboseExtended logging (debug level)off

Example: Running on a VPS

# Prepare config and certificates
mkdir -p /opt/co2h/certs /opt/co2h/logs
cp srv.toml /opt/co2h/
cp server.crt server.key ca.crt /opt/co2h/certs/

# Start in background
cd /opt/co2h
./teamserver --config srv.toml &

After starting, operators connect with the client by specifying the teamserver IP and port in the Connect to server dialog.
