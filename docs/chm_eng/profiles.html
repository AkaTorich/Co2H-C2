Malleable Profiles

A profile is a TOML file that describes how an HTTPS listener should "look" from the network perspective. Applies only to the HTTPS transport; TCP, SMB and DNS do not use profiles.

The profile is specified when creating a listener: the Profile field in the New Listener dialog, or the profile key in the [[listeners]] section of srv.toml. If the field is left empty, the built-in default profile is used.

Which Components Use the Profile

The profile is used in two places that must be in sync:

ComponentWhat It Takes from the Profile
Teamserver (HTTPS listener) Routing of incoming requests: compares method and URI against verb, uri_checkin, uri_task, http-post.uri.
Locates the encrypted beacon blob in the metadata_cookie cookie.
Response formatting: status, Content-Type, wrap_prefix/suffix, extra headers.
Artifact Generator When generating, embeds into the beacon the values of uri_checkin, uri_task, http-post.uri, User-Agent and metadata_cookie — the beacon will use exactly these URIs and pass metadata in this cookie.

If a beacon was generated with one profile but the listener is running with another, the URI or cookie name will not match and the beacon will not receive any tasks.

The [server] and [server.headers] fields affect only the teamserver (appearance of HTTP responses). The beacon does not read them — it accepts the body as-is.

Field Reference

[http-get]

FieldTypeDefaultDescription
uri_checkinstring/search URI the beacon sends check-in requests to (with the metadata cookie).
uri_taskstring/api/feed URI for fetching tasks. Can be the same as uri_checkin.
verbstringGET HTTP method for check-in and poll.
metadata_cookiestringsid Cookie name in which the beacon passes encrypted metadata on each request.

[http-get.client_headers]

Arbitrary HTTP headers for beacon GET requests. Typical fields:

HeaderExample
User-Agent"Mozilla/5.0 (Windows NT 10.0; Win64; x64) ..."
Accept"text/html,application/xhtml+xml,*/*;q=0.8"
Accept-Language"en-US,en;q=0.9"
Accept-Encoding"gzip, deflate, br"
Sec-Fetch-*Chrome browser security headers

[http-post]

FieldTypeDefaultDescription
uristring/submit URI for submitting beacon output (command results, files).
verbstringPOST HTTP method for submit.

[http-post.client_headers]

Arbitrary HTTP headers for beacon POST requests. Typical fields:

HeaderExampleNote
Content-Type"application/octet-stream" Default. To masquerade as a JSON API: "application/json; charset=utf-8".
User-Agent"Mozilla/5.0 ..." Usually matches GET. If not specified, uses the value from [http-get.client_headers].
Accept"*/*" For plausibility when masquerading as an API.
X-Requested-With"XMLHttpRequest" Characteristic of AJAX requests to web applications.
Origin / Referer"https://example.com" Simulating a browser request.

[server]

FieldTypeDefaultDescription
statusint200 HTTP status code for responses with payload. Can be set to 206 (Partial Content) for plausibility.
content_typestringtext/html; charset=utf-8 Value of the Content-Type header in the response.
wrap_prefixstring"" String prepended before the payload in the response body. For example: "<html><body>".
wrap_suffixstring"" String appended after the payload in the response body. For example: "</body></html>".

[server.headers]

Additional HTTP response headers. Added to every response from the listener. The Content-Length and Connection headers are added automatically — you do not need to specify them.

ExampleEffect
"Server" = "nginx"Masquerade as nginx.
"Cache-Control" = "no-cache"Appearance of dynamic content.
"X-Frame-Options" = "SAMEORIGIN"Simulates security headers of a browser application.
"X-XSS-Protection" = "0"Characteristic of Google servers (gws).

Default Profile

profiles/default.toml — nginx + Chrome UA. Check-in at /search, tasks at /api/feed, submit at /submit. Cookie sid. Used by default in configs/srv.toml.

Example: Masquerading as Office 365

[http-get]
uri_checkin     = "/owa/auth/logon.aspx"
uri_task        = "/owa/service.svc"
verb            = "GET"
metadata_cookie = "ClientId"

[http-get.client_headers]
"User-Agent"      = "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36"
"Accept"          = "text/html,application/xhtml+xml,*/*;q=0.8"
"Accept-Language" = "en-US,en;q=0.9"

[http-post]
uri  = "/owa/service.svc/s/GetMail"
verb = "POST"

[http-post.client_headers]
"Content-Type" = "application/json; charset=utf-8"
"X-OWA-CANARY" = "placeholder"

[server]
status       = 200
content_type = "application/json; charset=utf-8"
wrap_prefix  = ""
wrap_suffix  = ""

[server.headers]
"Server"        = "Microsoft-IIS/10.0"
"X-Powered-By"  = "ASP.NET"
"Cache-Control" = "private"
Note on wrap_prefix / wrap_suffix. When the beacon receives a response, it extracts base64url data from the entire body, ignoring characters outside the alphabet (A–Z a–z 0–9 - _). Therefore prefix and suffix must consist only of characters outside this alphabet: < > ! " { } : , / etc. If the prefix contains an alphanumeric character, a dash, or an underscore — it will be included in the decoded stream and the payload will be corrupted.
