Listeners

HTTPS

The primary listener for the C2 channel. Accepts three URIs (configurable in the malleable profile):

URIMethodPurpose
/search (default)GET Beacon check-in.
/api/feedGET Fetching tasks.
/submitPOST Submitting results.

Created via UI: Listeners → New → Kind: HTTPS. Specify host, port, URI paths and an optional malleable profile. The listener starts immediately and appears in the Listeners table.
Alternative: a static [[listeners]] entry in srv.toml — starts automatically when the teamserver starts.

TCP (raw)

The beacon connects to the teamserver directly over TCP without an HTTP wrapper. Useful for internal networks where HTTPS traffic is filtered or where a smaller signature is needed.

Created dynamically via UI: Listeners → New → Kind: TCP (raw). Specify the bind host and port; the key is generated by the server and shown in the Listeners tab.

The transport is selected automatically by the beacon: if the check-in address begins with tcp:// — this listener is used.

SMB (named pipe)

The beacon communicates via Windows named pipes — for lateral movement inside the network without going to the internet. Works on Windows only.

Created via UI: Listeners → New → Kind: SMB (named pipe). Specify the pipe name (default co2h).

The beacon selects the SMB transport if the check-in address begins with smb://. When generating an artifact from the GUI this is set automatically.

DNS (UDP)

The beacon sends data via DNS A-record queries and receives tasks via TXT records. All traffic is disguised as legitimate DNS queries; the teamserver acts as the authoritative NS server for a dedicated subdomain.

Created via UI: Listeners → New listener → Kind: DNS (UDP). Specify:

ParameterDescription
Bind hostAddress on which the teamserver listens for UDP (usually 0.0.0.0)
PortUDP port (standard — 53)
C2 domainAuthoritative subdomain, e.g. c2.example.com. The domain's DNS must have an NS record pointing to the teamserver's IP.

The transport is selected automatically by the beacon: if the check-in address begins with dns:// — the DNS transport is used. When generating an artifact from the GUI this is set automatically.

Protocol

  • Upload (beacon → server): A-queries in the form <d0_60hex>.<d1_60hex>.<type><seq><txid>.<domain> — 60 bytes of payload per query.
  • Upload completion: fin.<cnt>.<len>.<txid>.<domain>.
  • Receiving tasks (server → beacon): TXT query p.<txid>.<domain> — the response contains the encrypted task frame in hex encoding.

All frames are encrypted with AES-GCM using the listener key; the per-session key is wrapped with RSA-OAEP on the first check-in (same as HTTPS/TCP).

Pivot

A raw TCP channel between the beacon and teamserver — a "relay" for beacons in isolated network segments. Activated by the tcp_pivot <host> <port> command on the "head" beacon.

Created via UI: Listeners → New → Kind: Pivot (SOCKS5). Specify the bind host, SOCKS port and pivot port (must differ).

SOCKS5

Reverse proxy through the beacon: opens a local SOCKS5 port on the operator side, all traffic is tunneled through the beacon into the target network. Created via Sessions → right-click → Start SOCKS5.

Reverse Port Forward

The teamserver opens a TCP port and relays incoming connections through the specified beacon to rhost:rport. Created via the beacon context menu in Sessions, not through the Listeners tab.

ParameterValue
Bind hostAddress on which the teamserver will open the TCP port (0.0.0.0 for all interfaces)
Bind portPort on the teamserver to which external clients connect
RhostHost in the beacon's network to which it connects
RportPort on rhost
Beacon IDBound to a specific beacon; if the beacon is lost the listener closes

Fallback C2 (Automatic Channel Rotation)

A beacon can be configured to automatically switch between multiple C2 servers when the connection to the current one is lost. When generating an artifact, the Fallback C2 field accepts up to 4 fallback channels (one per line):

host:port
host:port:uri

Examples:

10.0.0.5:443
backup.example.com:8443
192.168.1.100:4444:tcp://
ns1.example.com:53:dns://c2.example.com

Switching Logic

  1. The beacon works with the primary channel.
  2. If 3 consecutive attempts (check-in or task fetch) fail — it switches to the next channel.
  3. Chain: primary → fb[0] → fb[1] → fb[2] → fb[3] → primary → ... (infinite loop).
  4. On the new channel, the beacon performs a new check-in.
  5. On the first successful response, the failure counter resets — the beacon stays on the current channel.

Requirement: Shared Key

All servers in the fallback chain must use the same listener key. Otherwise the beacon will not be able to check in on the fallback server.

How to set this up:

  1. Start the first server with a listener — the key will be generated automatically.
  2. Copy the key: Listeners tab → right-click → Copy listener key.
  3. In the fallback server's config (srv.toml) specify the copied key: 
    [[listeners]]
name         = "fallback"
kind         = "https"
bind         = "0.0.0.0"
port         = 443
listener_key = "paste_64_hex_chars_here"

If the listener_key field is not specified — the server generates a key automatically (normal mode without fallback).

Lifecycle

  1. HTTPS, TCP, SMB, DNS and Pivot types are created via the New listener button on the client toolbar or via the Add button in the Listeners tab. SOCKS5 and Reverse Port Forward are created via the beacon context menu (Sessions → right-click).
  2. Static alternative for HTTPS: a [[listeners]] section in srv.toml — the listener starts with the teamserver without any client interaction.
  3. The listener key is available in the Listeners tab (Key column) and is used when generating artifacts.
  4. To remove a listener — select the row and click Remove.
  5. The Listeners tab shows: name, type, bind address, key.
