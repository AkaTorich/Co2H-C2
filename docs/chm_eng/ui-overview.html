Certificate Generation

Menu Tools → Generate Certificates… — regenerate the full mTLS certificate set directly from the client, no external utilities required.

What is created in the certs/ folder next to the executable:

FilePurpose
ca.crt / ca.keySelf-signed root CA. Trust anchor for both sides of mTLS.
server.crt / server.keyTeamserver certificate. Signed by the CA.
listener.crt / listener.keyHTTPS listener certificate. Signed by the CA.
operator.crt / operator.keyOperator client certificate. Signed by the CA. This file is used in the login dialog.

Each generated certificate is validated against the chain before being written to disk. If validation fails — generation aborts, existing files are not overwritten, and an error message is written to the Logs tab.

If certificates already exist in certs/ — the client will ask for confirmation before overwriting. If the teamserver is running at the time of generation — it holds the old certificates in memory, so after successful generation the client will offer to restart it automatically. Without a restart the mTLS handshake will be rejected with an unknown CA error.
Generation takes about 30 seconds and runs in the background — the UI is not blocked, step-by-step progress is written to the Logs tab.

Shellcode Generation

Menu Tools → Generate Shellcode (scelot)… — dialog for converting an executable file or .NET assembly into position-independent shellcode.

FieldPurpose
Input PEPath to the source .exe / .dll on the operator's machine. The Browse… button opens a file selection dialog.
Output .binWhere to save the resulting shellcode. Defaults to the input filename with a .bin extension.
Architectureauto (from the input PE), x64, or x86.
Exit modeBehavior after the payload finishes: exit — terminate the process, thread — terminate the thread, return — return control.
ArgsCommand-line arguments passed to the payload.
Export (DLL)DLL export name to call after DllMain.
.NET class / .NET methodClass and method for .NET assemblies (Namespace.Class).
Also emit BOF stager (.o)Additionally build a BOF stager (COFF .o) next to the .bin. The generated shellcode is embedded in it; when loaded via bof <file.o> from another beacon it allocates RWX memory, copies the shellcode there and runs it in a new thread. No external compiler is required — a pre-compiled template is used, which scelot patches on the fly.

After clicking Generate, the output of scelot.exe is displayed in the lower log window of the dialog.

Plugin Bar

Between the toolbar and the main tabs there is a plugin bar — a row of icons for loaded plugins. Each icon acts as a toggle:

  • First click — opens the plugin interface (the main tabs are hidden).
  • Clicking the same icon again — hides the plugin and restores the client tabs.

Only one plugin can be active at a time. Clicking another plugin icon switches to it, and the previous one is hidden automatically.

Process Browser

The Processes tab — a tree view of processes on the remote machine with extended information. Core fields (memory, session, PPID) are populated without administrator privileges.

Control Panel

ElementDescription
Beacon (dropdown) Select the beacon from which processes are requested. Displayed as user@host (id). The list updates automatically when Sessions changes.
Refresh Send the ps command to the selected beacon and build the tree from the response.
Kill Send taskkill /F /PID <pid> via shell for the selected process. The button is active only when a row is selected.
Filter by name… Text filter. Searches by process name (Name column), user (User), and PID. Case-insensitive. Parent nodes remain visible if at least one child matches.

Tree Columns

ColumnAvailabilityDescription
PIDAlwaysProcess identifier. This column displays the tree structure with an orange dotted line and blue ± buttons.
PPIDAlwaysParent process PID. Used to build the hierarchy.
SessionAlwaysWindows session number (0 — services, 1+ — interactive).
ArchPrivilegedx64 or x86. If the beacon cannot open the process — ?.
MemoryAlwaysWorking Set. Displayed in MB or KB.
UserPrivilegedOwner: DOMAIN\user. If no access — -.

Columns PID, PPID, Session, Memory, and Name are populated for all processes regardless of the beacon's privilege level. Arch and User may be empty for processes belonging to other users. When running as administrator all processes are covered.

Tree Construction

The tree is built in two passes:

  1. Items are created for all processes from the response.
  2. Each item is attached as a child to the item whose PID equals its PPID. If the parent is not found — the process is placed at the root of the tree.

After construction the tree is fully expanded. Sorting is available by clicking any column header.
