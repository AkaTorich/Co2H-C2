Artifact Generator

An artifact is a beacon template with the C2 address and listener key already embedded. Two target OSes and four transports are supported.

Templates by Target OS

All templates reside in the bin\beacons\ directory alongside the client and artifact-gen. Template selection is fully automatic — just specify the target OS, format and architecture in the generation dialog.

A total of 16 templates for three OSes. All are accessible through the artifact generation dialog in the client (Toolbar → Generate artifact): selecting the OS automatically updates the format and architecture lists.

Generation is fully integrated into the client — no separate commands or utilities need to be run. For more on the dialog, see: Artifact Generation Dialog.

Creating from the Client

Toolbar → Generate artifact:

  1. In the Listener dropdown, select the desired listener — the key and parameters are filled in automatically.
  2. Fields change based on listener type:
    • HTTPS — C2 host and port.
    • TCP — C2 host and port; the beacon will select TCP transport automatically.
    • SMB — pipe name (taken from the listener automatically) and Server host — IP or hostname of the machine whose pipe the beacon will connect to.
    • DNS — C2 domain field (filled from the listener automatically). The C2 host field is hidden — the teamserver IP is taken automatically from the current connection address.
  3. (Optional) In the Fallback C2 field, specify fallback servers (up to 4, one per line, format host:port[:uri]). See Fallback C2.
  4. In the Target OS field, select Windows, Linux or macOS.
  5. Select format and architecture:
    • Windows — EXE / DLL, x64 / x86.
    • Linux — ELF / SO, x64 / ARM64 / ARM32.
    • macOS — Mach-O / DYLIB, Universal (arm64 + x64) / ARM64 / x64.
  6. Specify the save path and click OK. Progress and messages appear in the Logs tab.

Generating a Relay Child Beacon

Beacon context menu (Sessions or Graph) → Generate relay child beacon… opens the child artifact generation dialog, configured to connect through the selected parent beacon.

FieldDescription
Target OSTarget OS for the child beacon: Windows or Linux.
FormatEXE or DLL (Windows only; hidden when Linux is selected).
Architecturex64 or x86 (Windows only; hidden when Linux is selected).
OutputFile save path. When Linux is selected, the .exe/.dll extension is automatically removed.

The artifact is generated with these parameters:

  • Host — IP of the parent beacon (internal target address).
  • Port — relay port opened on the parent beacon.
  • Parent ID — hex identifier of the parent beacon (patched into the parent_id field of the BeaconState struct).
  • Transport — raw TCP (tcp:// in uri_checkin).
Before generating a child artifact, you must open a relay port on the parent beacon: context menu → Start relay port (chain pivot)…. The child beacon connects to this port and gains access to the teamserver through the relay chain.

DNS Artifact Requirements

For DNS C2 to work, you need to:

  1. Delegate a subdomain (e.g. c2.example.com) to the teamserver IP via an NS record at the registrar.
  2. Open UDP port 53 on the teamserver firewall.
  3. Create a DNS listener via New listener → DNS (UDP), specifying the same subdomain in the C2 domain field.
  4. Generate the artifact selecting that DNS listener. The C2 domain field will be filled in automatically.
