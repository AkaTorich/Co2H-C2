Teamserver

Teamserver запускается из клиента кнопкой Start Server на toolbar'е. Конфигурация (TLS-сертификаты, listener'ы, БД) хранится в configs/srv.toml рядом с клиентом и редактируется через GUI; в норме напрямую запускать teamserver.exe не требуется.

Структура конфига srv.toml

Общие секции

[operators]
bind = "0.0.0.0"
port = 50050

[tls]
cert = "certs/server.crt"
key  = "certs/server.key"
ca   = "certs/ca.crt"      # закомментировать для отключения mTLS (только для разработки)

[database]
path = "co2h.db"

[io]
threads = 0                 # 0 = авто (по числу ядер)

[log]
dir = "logs"

Конфигурация listener'ов

Каждый listener описывается секцией [[listeners]]. Можно указать несколько секций — все запустятся при старте teamserver'а.

HTTPS listener

[[listeners]]
kind         = "https"
name         = "https-main"
bind         = "0.0.0.0"
port         = 443
tls_cert     = "certs/listener.crt"
tls_key      = "certs/listener.key"
profile      = "profiles/default.toml"   # malleable-профиль (необязательно)
# listener_key = "aabb...64 hex символа" # необязательно; без него — авто
ПараметрОписаниеПо умолчанию
kindТип listener'а"https"
nameУникальное имя—
bindАдрес привязки"0.0.0.0"
portTCP-порт443
tls_certПуть к TLS-сертификату listener'а—
tls_keyПуть к приватному ключу TLS—
profileПуть к malleable-профилю (URI, заголовки, cookie)встроенный
listener_keyAES-256 ключ (64 hex). Пустой = авто из БД или рандомавто

TCP (raw) listener

TCP-листенеры обычно создаются динамически через UI клиента. При необходимости можно описать в конфиге:

[[listeners]]
kind         = "tcp"
name         = "tcp-internal"
bind         = "0.0.0.0"
port         = 4444
# listener_key = "aabb...64 hex символа"

SMB (named pipe) listener

SMB-листенеры также создаются динамически. Пример для конфига:

[[listeners]]
kind         = "smb"
name         = "smb-lateral"
bind         = "0.0.0.0"
port         = 445
# listener_key = "aabb...64 hex символа"

DNS (UDP) listener

[[listeners]]
kind         = "dns"
name         = "dns-c2"
bind         = "0.0.0.0"
port         = 53
c2_domain    = "c2.example.com"
# listener_key = "aabb...64 hex символа"
ПараметрОписание
c2_domainАвторитарный поддомен, делегированный на IP teamserver'а через NS-запись
portUDP-порт (стандартный — 53)

Несколько listener'ов одновременно

Можно запустить сразу HTTPS и DNS — каждый со своим ключом:

[[listeners]]
kind         = "https"
name         = "https-main"
bind         = "0.0.0.0"
port         = 443
tls_cert     = "certs/listener.crt"
tls_key      = "certs/listener.key"

[[listeners]]
kind         = "dns"
name         = "dns-backup"
bind         = "0.0.0.0"
port         = 53
c2_domain    = "c2.example.com"

Параметр listener_key

AES-256 ключ listener'а (64 hex-символа). Логика выбора ключа при старте:

  1. Если listener_key указан в конфиге — используется как есть и сохраняется в БД.
  2. Иначе — ключ берётся из БД (сохранён от предыдущего запуска).
  3. Если в БД нет — генерируется случайный и сохраняется.

Применение: когда beacon настроен на автоматическое переключение между серверами (fallback C2), все серверы в цепочке должны использовать один и тот же listener key.

Скопировать ключ текущего listener'а: вкладка Listeners → правый клик → Copy listener key.

Что делает teamserver при старте

  1. Читает конфиг и инициализирует подсистему сетевого ввода-вывода.
  2. Открывает базу данных и мигрирует схему.
  3. Поднимает порт для подключения операторов (mTLS).
  4. Запускает все listener'ы, объявленные в секциях [[listeners]].
  5. Генерирует ключ для каждого listener'а и отображает его во вкладке Listeners клиента.

Аутентификация операторов

Пароли операторов хранятся в БД в виде хеша. Дефолтный admin создаётся при первой инициализации:

usernameadmin
passwordchangeme
Сменить пароль — обязательная первая операция в реальном развёртывании.

Credentials store

Общий склад учётных данных всех операторов. Хранится в БД teamserver'а, изменения автоматически рассылаются всем подключённым операторам — вкладка Credentials у каждого обновляется без перезагрузки.

Поддерживаемые типы записей: password, hash, ticket, identity. Управление — через вкладку Credentials в клиенте: Refresh, Add, Delete.

Запуск из клиента

Toolbar → Start Server. Stdout/stderr процесса автоматически идут во вкладку Logs, статус (running / stopped) и порт listener'а отображаются в строке состояния клиента. Stop Server корректно завершает процесс.

Автономный запуск (без клиента)

Teamserver можно запустить отдельно от клиента — например, на удалённом VPS или в Docker-контейнере. Единственный обязательный аргумент — путь к конфигу:

Windows

teamserver.exe --config C:\path\to\srv.toml

Linux

./teamserver --config /opt/co2h/srv.toml

Если аргумент --config не указан, teamserver ищет configs/srv.toml в текущем рабочем каталоге.

Дополнительные аргументы

АргументОписаниеПо умолчанию
--config <path>Путь к TOML-конфигуconfigs/srv.toml
--bind <addr>Адрес привязки порта операторов (переопределяет конфиг)из конфига
--port <num>Порт для подключения операторов (переопределяет конфиг)из конфига
--verboseРасширенное логирование (debug-уровень)выкл.

Пример: запуск на VPS

# Подготовка конфига и сертификатов
mkdir -p /opt/co2h/certs /opt/co2h/logs
cp srv.toml /opt/co2h/
cp server.crt server.key ca.crt /opt/co2h/certs/

# Запуск в фоне
cd /opt/co2h
./teamserver --config srv.toml &

После запуска операторы подключаются клиентом, указав IP и порт teamserver'а в окне Connect to server.
