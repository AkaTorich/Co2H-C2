Команды Linux beacon'а

Linux beacon поддерживает все общие команды (файловая система, shell, sleep, exit, screenshot, download, upload) и набор команд, специфичных для Linux. Общие команды описаны в основном справочнике.

Linux beacon собирается как ELF-бинарник (beacon-linux64) и разделяемая библиотека (beacon-linux64.so). Библиотечный вариант запускается через LD_PRELOAD или dlopen — beacon стартует в отдельном потоке при загрузке .so.

Файловая система

КомандаАргументыОписание
cat<path> Вывести содержимое файла. Чанкованная передача для больших файлов.
mkdir<path> Создать каталог (mkdir(path, 0755)).
chmod<mode> <path> Сменить права файла. Режим задаётся восьмеричным числом (например chmod 755 /tmp/payload).

Выполнение и процессы

КомандаАргументыОписание
shell<cmd> Выполнить команду через fork() + execl("/bin/sh", "sh", "-c", cmd). Stdout и stderr перехватываются через pipe().
ps Список процессов: чтение /proc/[pid]/stat, /proc/[pid]/status и /proc/[pid]/cmdline. Формат: PID|PPID|UID|USER|CMD.
kill<pid> Завершить процесс сигналом SIGKILL.

Скриншот

КомандаАргументыОписание
screenshot Захват экрана в формате 24-bit BMP. Файл сохраняется на рабочем столе оператора: screenshot_<beaconId>_<timestamp>.bmp.

Два метода захвата (по приоритету):

  1. X11 — загрузка libX11.so.6 через dlopen() (без зависимости при компиляции). Захватывает корневое окно (XGetImage на root window). Поддержка 32/24/16 бит. Работает на системах с X-сервером.
  2. Фреймбуфер — чтение /dev/fb0 через mmap(). Работает на TTY, embedded-системах и серверах без графической оболочки. Требует доступ на чтение к /dev/fb0.
Если ни X11, ни фреймбуфер не доступны, команда возвращает ошибку: screenshot: no X11 display and /dev/fb0 unavailable.

Повышение привилегий

КомандаАргументыОписание
privesc_root[method] Повышение привилегий до root через уязвимости ядра.

Поддерживаются два семейства уязвимостей:

  • Dirty Pipe (CVE-2022-0847) — перезапись page cache через pipe buffer flag. Ядра 5.8–5.16.10.
  • Copy Fail (CVE-2026-31431) — scratch-write в page cache через authencesn шаблон AF_ALG. Ядра 4.14–6.19.12.

CVE-2026-31431 — Copy Fail

Логическая ошибка в шаблоне authencesn (Extended Sequence Number) модуля algif_aead. При расшифровке ядро записывает 4 байта из IV в позицию assoclen-8 исходного scatterlist'а. Если страницы данных были получены через splice() из файла, запись попадает напрямую в page cache.

Механизм:

  1. socket(AF_ALG) → bind к authencesn(hmac(sha256),cbc(aes))
  2. setsockopt: ключ (formат authencesn) + authsize=32
  3. accept() → operation fd
  4. sendmsg() с cmsg: op=decrypt, IV[0..3]=payload, assoclen=offset+12
  5. splice() страниц целевого файла в operation fd (zero-copy)
  6. recv() → ядро вызывает crypto_authenc_esn_decrypt() → запись IV[0..3] в src[assoclen-8] = page cache файла

Преимущества перед Dirty Pipe:

  • Работает на ядрах 4.14+ (Dirty Pipe: только 5.8–5.16.10)
  • Может писать по смещению 0 (Dirty Pipe: минимум 1)
  • Детерминистическая: нет гонки с pipe buffer flags

Ограничения:

  • 4 байта за операцию (Dirty Pipe: до PAGE_SIZE)
  • Требует модуль algif_aead + authencesn в ядре
  • Целевой файл должен быть ≥ 256 байт (для splice)

Оба метода записывают в page cache ядра — антивирусы и EDR на уровне пользователя не перехватывают запись. Однако целостность файлов (AIDE, dm-verity) может обнаружить изменения. Copy Fail оставляет следы в виде множества AF_ALG сокетов в аудит-логах (если включён audit).
Версия ядра: перед использованием проверьте версию командой shell uname -r. Copy Fail дополнительно проверяет доступность модуля authencesn — если модуль не загружен, вернёт ошибку.

Интерактивная оболочка (ishell)

КомандаАргументыОписание
ishell<text> Отправить строку на STDIN запущенного /bin/sh. Если оболочка ещё не создана — запускается автоматически (fork() + execl("/bin/sh"), ввод/вывод через socketpair(AF_UNIX)).
ishell(пустой) Закрыть интерактивную оболочку: закрыть сокет, дождаться завершения дочернего процесса.

В отличие от shell, который создаёт новый процесс на каждую команду, ishell держит один /bin/sh между вызовами. Это позволяет запускать длительные команды (linpeas, tcpdump и т.д.) и получать вывод по частям: основной цикл beacon'а вызывает ishell_pump() каждую итерацию и отправляет накопленный stdout/stderr.

Консоль оператора переключается в режим [interactive shell]. Каждая введённая строка — это ishell <text>. Команда exit внутри режима закрывает /bin/sh, не останавливая beacon.

Pivot и ретрансляция

КомандаАргументыОписание
tcp_pivot<host> <port> Открыть raw-TCP-канал к pivot-listener'у на teamserver'е. Beacon создаёт TCP-соединение к host:port и мультиплексирует трафик через этот канал. Позволяет beacon'у в изолированном сегменте достучаться до teamserver'а через beacon-посредника.

Relay (цепочный pivot)

Linux beacon поддерживает relay-механизм — родительский beacon открывает TCP-порт, к которому подключаются дочерние beacon'ы. Весь трафик дочерних beacon'ов ретранслируется через родительский к teamserver'у.

КомандаАргументыОписание
relay_start<port> Открыть TCP-listener на указанном порту. Каждое принятое соединение — дочерний beacon. Максимум 8 одновременных дочерних сессий.
relay_stop Закрыть relay-listener и все дочерние соединения.

Relay-кадры от дочерних beacon'ов отправляются с магическим task_id = 0xFFFFFFFFFFFFFFFD, который teamserver маршрутизирует отдельно от обычных ответов. Оператор видит дочерние beacon'ы как независимые сессии в таблице Sessions.

Relay-команды вызываются из контекстного меню Sessions/Graph: Start relay port (chain pivot)… и Generate relay child beacon…. Вручную из консоли вводить не нужно.

Reverse Port Forward

Teamserver открывает TCP-порт (bind); при входящем подключении трафик идёт через C2-канал к beacon'у, который пробрасывает его к rhost:rport. Архитектурно — зеркало SOCKS5: клиент подключается к teamserver'у, а не к прокси-клиенту оператора. Максимум 16 одновременных соединений.

Способ созданияОписание
Sessions → правый клик → Start Reverse Port Forward… Диалог: Bind host и Bind port — что откроет teamserver; Rhost и Rport — цель, к которой подключается beacon. После нажатия OK teamserver начинает слушать. При первом входящем подключении beacon устанавливает соединение с rhost:rport.

Схема трафика:

внешний клиент → teamserver:bind_port → (C2-канал) → beacon → rhost:rport

Протокол: данные передаются через out-of-band кадры с магическим task_id = 0xFFFFFFFFFFFFFFFC (RPORTFWD_TASK_MAGIC). Формат кадра: [u64 conn_id BE][u8 type][payload], где type:

  • 0x01 — CONNECT_OK (beacon успешно подключился к rhost:rport)
  • 0x02 — CONNECT_FAIL (ошибка подключения)
  • 0x03 — DATA: [u32 len BE][bytes]
  • 0x04 — CLOSE (соединение закрыто)
Пример — доступ к внутреннему веб-серверу через teamserver:
Bind host: 0.0.0.0, Bind port: 8080, Rhost: internal-webapp.corp.local, Rport: 80
curl http://teamserver-ip:8080/

Отличие от SOCKS5: SOCKS5 туннелирует произвольный трафик через polling-цикл; rportfwd открывает конкретный порт напрямую на teamserver'е, не требует SOCKS5-клиента, работает для любого TCP-протокола.
