Интерфейс оператора

Главное окно состоит из пяти логических частей:

  1. Меню (File / View / Help)
  2. Панель инструментов (toolbar)
  3. Панель плагинов (иконки загруженных плагинов)
  4. Вкладки: Sessions / Listeners / Graph / Files / Processes / Downloads / Credentials / Chat / Logs
  5. Консоль beacon'а (внизу, разделитель сверху)

Генерация сертификатов

Меню Tools → Generate Certificates… — регенерация всего комплекта mTLS-сертификатов прямо из клиента, никаких внешних утилит ставить не нужно.

Что создаётся в папке certs/ рядом с исполняемым файлом:

ФайлНазначение
ca.crt / ca.keyСамоподписанный корневой УЦ. Trust anchor для обеих сторон mTLS.
server.crt / server.keyСертификат teamserver'а. Подписан CA.
listener.crt / listener.keyСертификат HTTPS-listener'а. Подписан CA.
operator.crt / operator.keyКлиентский сертификат оператора. Подписан CA. Этот файл подставляется в окно логина.

Каждый созданный сертификат проверяется на корректность цепочки до записи на диск. Если проверка не проходит — генерация прерывается, существующие файлы не перезаписываются, во вкладке Logs выводится сообщение об ошибке.

Если сертификаты в certs/ уже есть — клиент спросит подтверждение перезаписи. Если teamserver запущен в момент генерации — он держит старые сертификаты в памяти, поэтому после успешной генерации клиент предложит автоматически его перезапустить. Без перезапуска mTLS-рукопожатие будет отвергнуто с ошибкой неизвестного УЦ.
Генерация занимает около 30 секунд и идёт в фоне — UI не блокируется, прогресс по шагам пишется во вкладку Logs.

Генерация шеллкода

Меню Tools → Generate Shellcode (scelot)… — диалог преобразования исполняемого файла или .NET-сборки в позиционно-независимый шеллкод.

ПолеНазначение
Input PEПуть к исходному .exe / .dll на машине оператора. Кнопка Browse… открывает диалог выбора файла.
Output .binКуда сохранить готовый шеллкод. По умолчанию подставляется имя входного файла с расширением .bin.
Architectureauto (по входному PE), x64 или x86.
Exit modeПоведение после завершения нагрузки: exit — завершить процесс, thread — завершить поток, return — вернуть управление.
ArgsАргументы командной строки, передаваемые нагрузке.
Export (DLL)Имя экспорта DLL для вызова после DllMain.
.NET class / .NET methodКласс и метод для .NET-сборок (Namespace.Class).
Also emit BOF stager (.o)Дополнительно собрать BOF-стейджер (COFF .o) рядом с .bin. В нём зашит сгенерированный шеллкод; при загрузке через bof <файл.o> со стороны другого beacon'а он аллоцирует RWX-память, копирует туда шеллкод и запускает его в новом потоке. Внешний компилятор не требуется — используется заранее скомпилированный шаблон, который scelot патчит на лету.

После нажатия Generate вывод scelot.exe отображается в нижнем окне лога диалога.

Диалог генерации артефакта

Открывается кнопкой Generate artifact на панели инструментов или из контекстного меню listener'а.

ПолеОписание
ListenerВыбор listener'а из списка. Ключ и параметры подставляются автоматически.
Target OSЦелевая операционная система: Windows или Linux. Влияет на доступные форматы и шаблон beacon'а.
FormatФормат артефакта. Для Windows: EXE или DLL. При выборе Linux поле скрыто — всегда ELF.
ArchitectureАрхитектура: x64 или x86. При выборе Linux поле скрыто — только x64.
Fallback C2Резервные C2-каналы (до 4 штук, по одному на строку). Формат: host:port или host:port:uri. При потере связи с основным каналом beacon автоматически переключается на следующий. Подробнее — Fallback C2.
OutputПуть сохранения сгенерированного файла.

Поведение при выборе Target OS

WindowsLinux
Шаблонbeacon64.exe / beacon64.dll / beacon32.exe / beacon32.dll beacon-linux64 (ELF) или beacon-linux64.so
FormatEXE / DLLскрыт (ELF)
Architecturex64 / x86скрыт (x64)
Расширение.exe / .dll без расширения
Логика патчинга в artifact-gen одинакова для PE и ELF — сентинел (32 нулевых байта + UTF-16LE "127.0.0.1") ищется одинаково в обоих форматах. Аргумент --os linux передаётся для выбора шаблонного бинарника.

Панель плагинов

Между панелью инструментов и основными вкладками расположена панель плагинов — ряд иконок загруженных плагинов. Каждая иконка работает как переключатель:

  • Первый клик — открывает интерфейс плагина (основные вкладки скрываются).
  • Повторный клик по той же иконке — скрывает плагин и возвращает вкладки клиента.

Одновременно может быть активен только один плагин. Клик по иконке другого плагина переключает на него, предыдущий скрывается автоматически.

Вкладка Listeners

Список запущенных listener'ов teamserver'а: имя, тип (https/tcp/smb/pivot/socks5), адрес, порт. Создание listener'ов — через кнопку New listener на панели инструментов или RPC напрямую.

Вкладка Graph

Визуальная карта инфраструктуры в реальном времени. Обновляется автоматически при появлении, исчезновении или изменении beacon'ов и listener'ов.

Вкладка Downloads

Список файлов, скачанных с целей командой download. Показывает имя файла, размер, beacon-источник, время получения. Двойной клик — сохранить локально.

Вкладка Credentials

Общий склад учётных данных всех операторов. Хранится на teamserver'е, изменения автоматически рассылаются всем подключённым операторам.

КолонкаОписание
User / DomainИмя учётной записи
Kindpassword / hash / ticket / identity
SecretЗамаскирован в таблице; полный текст — в tooltip
Host / Source / ByОткуда получен

Кнопки: Refresh — запросить список с сервера; Add — добавить вручную; Delete — удалить выбранную строку.
getuid автоматически добавляет запись kind=identity при каждом выполнении.

Вкладка Chat

Командный чат: все операторы в одном канале. При получении нового сообщения в неактивной вкладке появляется маркер • в заголовке вкладки.

Вкладка Logs

Stdout/stderr запущенного teamserver'а + локальные сообщения клиента (подключение, генерация артефакта и т. п.). При новых строках в фоне заголовок вкладки помечается маркером •.

Обозреватель процессов

Вкладка Processes — древовидный список процессов на удалённой машине с расширенной информацией. Основные поля (память, сессия, PPID) заполняются без привилегий администратора.

КолонкаДоступностьОписание

Колонки PID, PPID, Session, Memory и Name заполняются для всех процессов независимо от уровня привилегий beacon'а. Arch и User могут быть пустыми для процессов других пользователей. При запуске от имени администратора покрываются все процессы.

Построение дерева

Дерево строится в два прохода:

  1. Создаются элементы для всех процессов из ответа.
  2. Каждый элемент привязывается как дочерний к элементу с PID = PPID. Если родитель не найден — процесс помещается в корень дерева.

После построения дерево полностью раскрывается. Сортировка доступна кликом по заголовку любой колонки.
