Listener'ы

HTTPS

Базовый listener для C2-канала. Принимает три URI (настраиваются в malleable-профиле):

URIМетодНазначение
/search (по умолчанию)GET Check-in beacon'а.
/api/feedGET Получение задач.
/submitPOST Сабмит результата.

Создаётся через UI: Listeners → New → Kind: HTTPS. Указать host, port, URI-пути и опциональный malleable-профиль. Listener запускается немедленно и отображается в таблице Listeners.
Альтернатива: статическая запись [[listeners]] в srv.toml — запускается при старте teamserver'а автоматически.

TCP (raw)

Beacon подключается к teamserver'у напрямую по TCP без HTTP-обёртки. Удобен для внутренней сети где HTTPS-трафик фильтруется или где нужна меньшая сигнатура.

Создаётся динамически через UI: Listeners → New → Kind: TCP (raw). Указать bind host и порт; ключ генерируется сервером и отображается во вкладке Listeners.

Транспорт выбирается beacon'ом автоматически: если адрес check-in начинается с tcp:// — используется этот listener.

SMB (named pipe)

Beacon общается через именованный канал Windows — для lateral movement внутри сети без выхода в интернет. Работает только на Windows.

Создаётся через UI: Listeners → New → Kind: SMB (named pipe). Указать имя pipe (по умолчанию co2h).

Beacon выбирает SMB-транспорт если адрес check-in начинается с smb://. При генерации артефакта из GUI это выставляется автоматически.

DNS (UDP)

Beacon отправляет данные через DNS-запросы типа A и получает задачи через TXT-записи. Весь трафик маскируется под легитимные DNS-запросы; teamserver выступает авторитарным NS-сервером для выделенного поддомена.

Создаётся через UI: Listeners → New listener → Kind: DNS (UDP). Указать:

ПараметрОписание
Bind hostАдрес, на котором teamserver слушает UDP (обычно 0.0.0.0)
PortUDP-порт (стандартный — 53)
C2 domainАвторитарный поддомен, например c2.example.com. В DNS-записях домена должна стоять NS-запись на IP teamserver'а.

Транспорт выбирается beacon'ом автоматически: если адрес check-in начинается с dns:// — используется DNS-транспорт. При генерации артефакта из GUI это выставляется автоматически.

Протокол

  • Загрузка (beacon → сервер): A-запросы вида <d0_60hex>.<d1_60hex>.<type><seq><txid>.<domain> — 60 байт полезной нагрузки на запрос.
  • Завершение загрузки: fin.<cnt>.<len>.<txid>.<domain>.
  • Получение задач (сервер → beacon): TXT-запрос p.<txid>.<domain> — ответ содержит зашифрованный фрейм задач в hex-кодировке.

Все фреймы шифруются AES-GCM с listener-ключом; per-session ключ оборачивается RSA-OAEP при первом check-in (аналогично HTTPS/TCP).

Pivot

Raw TCP-канал между beacon'ом и teamserver'ом — «ретранслятор» для beacon'ов в изолированных сегментах сети. Активируется командой tcp_pivot <host> <port> на «головном» beacon'е.

Создаётся через UI: Listeners → New → Kind: Pivot (SOCKS5). Указать bind host, SOCKS-порт и pivot-порт (должны отличаться).

SOCKS5

Reverse proxy через beacon: открывает локальный SOCKS5-порт на стороне оператора, весь трафик туннелируется через beacon в целевую сеть. Создаётся через Sessions → правый клик → Start SOCKS5.

Reverse Port Forward

Teamserver открывает TCP-порт и ретранслирует входящие соединения через указанный beacon к rhost:rport. Создаётся через контекстное меню beacon'а в Sessions, а не через вкладку Listeners.

ПараметрЗначение
Bind hostАдрес, на котором teamserver откроет TCP-порт (0.0.0.0 для всех интерфейсов)
Bind portПорт на teamserver'е, к которому подключается внешний клиент
RhostХост в сети beacon'а, к которому он подключается
RportПорт на rhost
Beacon IDПривязан к конкретному beacon'у; при потере beacon'а listener закрывается

Fallback C2 (автоматическая ротация каналов)

Beacon может быть настроен на автоматическое переключение между несколькими C2-серверами при потере связи с текущим. При генерации артефакта в поле Fallback C2 указываются до 4 резервных каналов (по одному на строку):

host:port
host:port:uri

Примеры:

10.0.0.5:443
backup.example.com:8443
192.168.1.100:4444:tcp://
ns1.example.com:53:dns://c2.example.com

Логика переключения

  1. Beacon работает с основным каналом.
  2. Если 3 попытки подряд (check-in или получение задач) провалились — переключается на следующий канал.
  3. Цепочка: основной → fb[0] → fb[1] → fb[2] → fb[3] → основной → ... (бесконечный цикл).
  4. На новом канале beacon выполняет повторный check-in.
  5. При первом успешном ответе счётчик неудач сбрасывается — beacon остаётся на текущем канале.

Требование: общий ключ

Все серверы в цепочке fallback должны использовать один и тот же listener key. Иначе beacon не сможет пройти check-in на резервном сервере.

Как настроить:

  1. Запустить первый сервер с listener'ом — ключ сгенерируется автоматически.
  2. Скопировать ключ: вкладка Listeners → правый клик → Copy listener key.
  3. В конфиге резервного сервера (srv.toml) указать скопированный ключ: 
    [[listeners]]
name         = "fallback"
kind         = "https"
bind         = "0.0.0.0"
port         = 443
listener_key = "вставить_64_hex_символа"

Если поле listener_key не указано — сервер генерирует ключ самостоятельно (обычный режим работы без fallback).

Жизненный цикл

  1. Типы HTTPS, TCP, SMB, DNS и Pivot создаются через кнопку New listener в toolbar'е клиента или через кнопку Add во вкладке Listeners. SOCKS5 и Reverse Port Forward создаются через контекстное меню beacon'а (Sessions → правый клик).
  2. Статическая альтернатива для HTTPS: секция [[listeners]] в srv.toml — listener стартует вместе с teamserver'ом без участия клиента.
  3. Ключ listener'а доступен во вкладке Listeners (колонка Key) и используется при генерации артефактов.
  4. Для удаления listener'а — выбрать строку и нажать Remove.
  5. Вкладка Listeners показывает: имя, тип, адрес привязки, ключ.
