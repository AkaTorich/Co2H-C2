Malleable-профили

Профиль — TOML-файл, описывающий, как HTTPS-listener должен «выглядеть» с точки зрения сети. Применяется только к HTTPS-transport'у; TCP, SMB и DNS профиль не используют.

Профиль задаётся при создании listener'а: поле Profile в диалоге New Listener, либо ключ profile в секции [[listeners]] файла srv.toml. Если поле оставить пустым — применяется встроенный профиль по умолчанию.

Какие компоненты используют профиль

Профиль задействован в двух местах, которые должны быть согласованы:

КомпонентЧто берёт из профиля
Teamserver (HTTPS listener) Маршрутизация входящих запросов: сравнивает метод и URI с полями verb, uri_checkin, uri_task, http-post.uri.
Поиск зашифрованного blob beacon'а в куке metadata_cookie.
Формирование ответа: статус, Content-Type, wrap_prefix/suffix, дополнительные заголовки.
Генератор артефактов При генерации вшивает в beacon значения uri_checkin, uri_task, http-post.uri, User-Agent и metadata_cookie — beacon будет обращаться именно на эти URI и передавать метаданные именно в этой куке.

Если beacon сгенерирован под один профиль, а listener запущен с другим — URI или имя куки не совпадут и beacon не получит ни одной задачи.

Поля [server] и [server.headers] влияют только на teamserver (вид HTTP-ответов). Beacon их не читает — он принимает тело как есть.

Справочник полей

[http-get]

ПолеТипПо умолчаниюОписание
uri_checkinstring/search URI, на который beacon шлёт check-in (с кукой метаданных).
uri_taskstring/api/feed URI для получения задач. Может совпадать с uri_checkin.
verbstringGET HTTP-метод для check-in и poll.
metadata_cookiestringsid Имя куки, в которой beacon передаёт зашифрованные метаданные при каждом обращении.

[http-get.client_headers]

Произвольные HTTP-заголовки GET-запросов beacon'а. Типичные поля:

ЗаголовокПример
User-Agent"Mozilla/5.0 (Windows NT 10.0; Win64; x64) ..."
Accept"text/html,application/xhtml+xml,*/*;q=0.8"
Accept-Language"en-US,en;q=0.9"
Accept-Encoding"gzip, deflate, br"
Sec-Fetch-*браузерные security-заголовки Chrome

[http-post]

ПолеТипПо умолчаниюОписание
uristring/submit URI для отправки вывода beacon'а (результаты команд, файлы).
verbstringPOST HTTP-метод submit.

[http-post.client_headers]

Произвольные HTTP-заголовки POST-запросов beacon'а. Типичные поля:

ЗаголовокПримерПримечание
Content-Type"application/octet-stream" По умолчанию. Для маскировки под JSON-API: "application/json; charset=utf-8".
User-Agent"Mozilla/5.0 ..." Обычно совпадает с GET. Если не указан — использует значение из [http-get.client_headers].
Accept"*/*" Для правдоподобности при маскировке под API.
X-Requested-With"XMLHttpRequest" Характерно для AJAX-запросов к веб-приложениям.
Origin / Referer"https://example.com" Имитация запроса из браузера.

[server]

ПолеТипПо умолчаниюОписание
statusint200 HTTP-статус ответов с payload. Можно поставить 206 (Partial Content) для правдоподобности.
content_typestringtext/html; charset=utf-8 Значение заголовка Content-Type в ответе.
wrap_prefixstring"" Строка, добавляемая до payload в теле ответа. Например: "<html><body>".
wrap_suffixstring"" Строка, добавляемая после payload в теле ответа. Например: "</body></html>".

[server.headers]

Дополнительные HTTP-заголовки ответа. Добавляются к каждому ответу listener'а. Заголовки Content-Length и Connection добавляются автоматически — указывать их не нужно.

ПримерЭффект
"Server" = "nginx"Маскировка под nginx.
"Cache-Control" = "no-cache"Вид динамического контента.
"X-Frame-Options" = "SAMEORIGIN"Имитация защитных заголовков браузерного приложения.
"X-XSS-Protection" = "0"Характерно для Google-серверов (gws).

Стандартный профиль

profiles/default.toml — nginx + Chrome UA. Check-in на /search, задачи на /api/feed, submit на /submit. Кука sid. Используется по умолчанию в configs/srv.toml.

Пример: маскировка под Office 365

[http-get]
uri_checkin     = "/owa/auth/logon.aspx"
uri_task        = "/owa/service.svc"
verb            = "GET"
metadata_cookie = "ClientId"

[http-get.client_headers]
"User-Agent"      = "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36"
"Accept"          = "text/html,application/xhtml+xml,*/*;q=0.8"
"Accept-Language" = "en-US,en;q=0.9"

[http-post]
uri  = "/owa/service.svc/s/GetMail"
verb = "POST"

[http-post.client_headers]
"Content-Type" = "application/json; charset=utf-8"
"X-OWA-CANARY" = "placeholder"

[server]
status       = 200
content_type = "application/json; charset=utf-8"
wrap_prefix  = ""
wrap_suffix  = ""

[server.headers]
"Server"        = "Microsoft-IIS/10.0"
"X-Powered-By"  = "ASP.NET"
"Cache-Control" = "private"
Замечание по wrap_prefix / wrap_suffix. Beacon при получении ответа извлекает base64url-данные из всего тела, игнорируя символы вне алфавита (A–Z a–z 0–9 - _). Поэтому prefix и suffix должны состоять только из символов вне этого алфавита: < > ! " { } : , / и т.п. Если prefix содержит буквенно-цифровой символ или дефис/подчёркивание — он будет включён в декодируемый поток и payload окажется повреждён.
